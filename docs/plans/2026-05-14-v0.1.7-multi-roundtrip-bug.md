# v0.1.7 Fix Prompt — Multi Round-Trip P&L Accuracy

**Status:** CLOSED 2026-05-21 — verified structurally absent in current code; see the Resolution block below. Pre-existing bug discovered during v0.1.6 smoke test on 2026-05-14. Shipped v0.1.6 as-is per user direction.

**Severity:** Critical. Any day with multiple round trips of the same symbol is under-charted in unfiltered Reports views — the smoking-gun example below misses 28% of true day P&L.

---

## Resolution — CLOSED 2026-05-21

Verified during the 2026-05-21 Day 8 audit: the bug is **structurally absent**
in current code. The v0.2.0 Day 1 universal-import rewrite (commit `3f1df7d`)
replaced the orphaned `electron/import/compute-trips.ts` path this doc flagged.
In the current path:

- `src/core/import/build-round-trips.ts:91-102` closes a round trip on every
  net-zero position return, so two same-symbol cycles in one day produce two
  distinct round trips.
- `electron/import/repo.ts:158` inserts with `ON CONFLICT(exec_hash) DO NOTHING`
  — the conflict key is `exec_hash`, not `(symbol, date)`, so a second same-day
  trip is never clobbered.
- `src/core/performance/metrics.ts:42-65` (`computeDailyPnL`) sums `net_pnl`
  keyed by `date` alone — it never groups by symbol.

Regression coverage added the same day:
`src/core/performance/__tests__/metrics.test.ts` implements this doc's "Test 2"
— the ODYS anchor case (+$17.69 + $6.78 = +$24.47 on 2026-05-11) plus
cross-symbol, multi-day, and win/loss variants across `computeDailyPnL`,
`computeCumulativePnL`, `computeDailyVolume`, and `computeDailyWinRate`. The
suite passes against current code.

This doc's "Test 1" (the round-trip builder) is already satisfied by
`src/core/import/__tests__/buildRoundTrips.test.ts` ("emits two round trips when
the same symbol cycles to flat twice"). "Test 3" (the Reports "round trips
analyzed" header count) is out of scope here — it belongs to the Reports IPC
layer and is not part of this verification.

Everything below this line is the original 2026-05-14 bug prompt, preserved
unchanged as the historical record.

## Reproduction (anchor case)

May 11 ODYS trading day from a real DAS execution log has TWO separate round trips:

| RT | Entry | Exit-to-flat | Net P&L | Hold |
|----|-------|--------------|---------|------|
| RT1 | 08:35:11 buy 25 @ $6.45 | 08:39:50 fully flat | **+$17.69** | 4m 39s |
| RT2 | 08:44:15 buy 25 @ $10.56 | 08:44:22 fully flat | **+$6.78** | 7 s |
| | | **True day P&L** | **+$24.47** | |

What FugaEdge currently shows on Reports for this day:

| Filter | Shown P&L | Expected | Verdict |
|--------|-----------|----------|---------|
| Float < 1M | +$6.78 | +$6.78 (RT2) | ✓ |
| Float 1–5M | +$17.69 | +$17.69 (RT1) | ✓ |
| ALL / unfiltered | **+$17.69** | **+$24.47** | **✗ missing RT2** |
| 90-day "Round trips analyzed" header | "3 round trips" | TBD — verify RT2 even reached DB | ? |

That the two per-float-bucket filters each return *one* trip suggests both RT1 and RT2 do exist in some queryable form — but the unfiltered aggregation is collapsing or losing one. The 90-day "3 round trips" count needs to be checked against `SELECT COUNT(*) FROM trades` so we know whether RT2 is in the DB at all.

## Where the bug lives (one of three)

Pick exactly one by evidence:

**(a) Merged on import** — `electron/import/parse-executions.ts` or `electron/import/compute-trips.ts` is collapsing RT1+RT2 into a single trip before it reaches the DB. Symptoms: only one row in `trades` for ODYS 2026-05-11.

**(b) Dropped on import** — same files, but RT2 is parsed and then discarded (e.g., a `LIMIT 1`, a `Map.set(symbol+date, ...)` overwrite, or a fee/scratch filter that nukes the smaller trip). Symptoms: same — one row in `trades` — but the executions table contains all fills.

**(c) Lost in aggregation** — both trips are in `trades` correctly, but Reports' unfiltered roll-up (`electron/reports/get.ts` and/or `src/core/performance/metrics.ts`) groups by `(symbol, date)` and keeps only one. Symptoms: two rows in `trades` for ODYS 2026-05-11, but Reports shows one.

The fact that per-float filters return each trip individually points hard at **(c)** — but verify, don't assume. If RT1 and RT2 land in different float-buckets, each bucket query naturally returns its single match; the bug may still be at the import layer.

   ```sql

     FROM trades
    WHERE symbol = 'ODYS' AND date = '2026-05-11'
    ORDER BY open_time;
   ```

   - If you get **1 row**: bug at import (a) — trips merged.
   - If you get **2 rows**: bug is downstream (c).

2. **If 2 rows in DB**, run the Reports daily aggregator against an in-memory `TradeListRow[]` containing exactly those two ODYS trips. The day total should be +$24.47. If it returns +$17.69, the merge is happening in `src/core/performance/metrics.ts`. Bisect from there.

If you only get 1, the bug is in `compute-trips.ts` (most likely a guard that breaks on rapid re-entry within seconds, e.g., the 7-second RT2).
