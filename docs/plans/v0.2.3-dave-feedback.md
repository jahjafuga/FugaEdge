# v0.2.3 — Dave feedback (country-data correctness)

Branch: `feature/v0.2.3-dave-feedback`.

## Stage 1 — FMP as primary country source (shipped, committed)

Polygon's free tier omits company domicile, so US-listed foreign issuers
defaulted to "US (inferred)" — the bug Dave hit. FMP `/stable/profile` returns
the real domicile as ISO 3166-1 alpha-2 (e.g. SPRC → IL, HTCO → SG), verified
across a 12-ticker basket on 2026-05-31.

- `src/services/fmp.ts` — `fetchCompanyProfile` (mirrors `fetchSharesFloat`).
- `src/core/country/source.ts` — reconciled source types: `ResolvedSource =
  'fmp' | 'polygon' | 'inferred' | 'unknown'`, `CountrySource = ResolvedSource
  | 'manual'`. `'fmp'` is a CONFIDENT source (not re-resolved on an incremental
  run, only on force) — same tier as `'polygon'`.
- `src/core/country/resolve.ts` — `resolveCountryFromFmp`.
- Orchestrator: FMP PRIMARY, Polygon FALLBACK (only on FMP-null). A confident
  FMP hit short-circuits Polygon. An FMP throw is recorded but still falls
  back. Country resolution behaviour is otherwise unchanged.
- New-import / on-demand path only. Existing-trade backfill
  (`backfillAllCountries`) stays Polygon-only — a separate gated step.

## Stage 2 — market_cap + sector + industry from the SAME FMP profile call

The country phase already fetches `/stable/profile` per imported symbol. Stage 2
reads three more fields from that SAME response — **ZERO extra requests**:

- `market_cap` — numeric (`toNullableNumber`; `''` → null, never 0).
- `sector`     — FMP taxonomy bucket, e.g. "Healthcare", "Technology".
- `industry`   — **new column** (schema 21 → 22, additive), more granular than
                 the sector column, e.g. "Biotechnology", "Semiconductors".

### TAXONOMY — important for future analytics

FMP `sector`/`industry` are **FMP's own clean buckets (e.g. "Healthcare", "Technology"), NOT Polygon's SIC-description
text**. Polygon returned things like `"PHARMACEUTICAL PREPARATIONS"`; FMP
returns sector `"Healthcare"` + industry `"Biotechnology"`. Any future
sector/industry breakdown, filter, or pillar logic must expect the FMP
vocabulary, not SEC SIC strings. (These three fields are currently write-only —
nothing in the UI/analytics reads them yet — so consumer-side risk is low.)

### Migration (Option A + version bump — additive, no module/latch/backup)

- `electron/db/schema.ts` — `industry TEXT` added to the `CREATE TABLE
  market_data` body (fresh installs); `SCHEMA_VERSION` 21 → 22 (release
  tracking).
- `electron/db/database.ts` — guarded `if (!hasMarket('industry')) ALTER TABLE
  market_data ADD COLUMN industry TEXT` in the existing `migrateAfterSchema`
  `hasMarket` block, mirroring country/region. Purely additive: a NULL column,
  no data transform, no backfill. Existing rows stay NULL until re-touched.

### Threading (Option B) + boundaries

- `import-orchestrator.ts` carries country + the three passengers together
  (`ResolvedSymbol`). `resolve-countries.ts` writes them to `market_data` in
  the country phase. A null passenger never wipes a prior value (prefer fresh
  FMP, else keep existing).
- The market-refresh path (`electron/market/fetch.ts`) STILL sources
  market_cap/sector from Polygon and writes `industry: null`. `upsertMarketRow`
  COALESCEs `industry` so a refresh never wipes an FMP-imported value.
- Out of scope (unchanged): existing-trade backfill, float / shares-float,
  retiring any Polygon call.

## Stage 3 — MAE/MFE sub-resolution hold filter (schema 24 → 25)

`computeMaeMfe` now returns `{mae: null, mfe: null}` for **closed** trades held
shorter than **α = 120s** (= 2× the 1-minute Polygon bar resolution): a sub-2-bar
scalp's entry/exit can land inside a single bar whose high/low reflect price the
trader never held through, inflating the excursion. Open trades (no `close_time`)
are unaffected — `holdMs` is non-finite, so the guard is skipped. Threshold is a
single source-of-truth constant (`MAE_MFE_MIN_HOLD_MS`, `electron/market/intraday.ts`);
all five read surfaces inherit the NULL via the stored `trades.mae/mfe` columns
(they already render "—"/"Awaiting" on null). Finer resolution (sub-minute /
tick excursion for scalps below α) is gated on the **v0.4.0+ webapp** tick/second-
bar data work — not solvable on the free-tier 1-minute Polygon feed.

Migration `electron/db/migrate-reset-mae-mfe.ts` (schema 24 → 25) nulls every
`trades.mae/mfe` (values computed under the old pre-α rule), then arms a
background recompute (`runPendingMaeMfeBackfill`, scheduled at `ready-to-show`)
off the cached `intraday_bars` — no new API calls. Unlike the additive Stage 2
migration this **destroys** values only recoverable while bar coverage holds, so
it takes a throw-to-abort pre-migration backup (house pattern: timestamped copy
in `userData/backups/`, WAL checkpoint + `-wal`/`-shm` sidecars, settings latch).
