# FugaEdge v0.2.4 — "Technical Analysis" Tab Specification

**Bottom line:** Build the Technical Analysis tab as a sibling tab at the top of the Analytics surface, driven by a pre-computed `trade_technicals` table (entry-only indicator state, 1M and 5M), with all six approved sections rendered in a hybrid layout (4 dashboard cards on top, full-width sectioned bands below with inline accordion drill-downs). The single most important design decision is **pre-compute, don't compute-on-query** — the four-bucket grid plus six sections over a 30–90 day window cannot recompute EMAs over thousands of warmup bars on every filter change. Everything else flows from that and from anchoring the feature on Ross Cameron's literal, verbatim rule: *"I only trade when the MACD is positive"* (Warrior Trading, "Diving Into The MACD Scalping Strategy"), which no competitor journal models at the per-trade entry-state level.

## TL;DR

- **Centerpiece feature, top-level placement:** a new Analytics sibling tab whose hero is a four-bucket MACD-state-at-entry grid (Positive+Rising / Positive+Falling / Negative+Rising / Negative+Falling) with win rate, net P&L, avg winner/loser, and expectancy per cell — backed by a stored `trade_technicals` table, not on-the-fly computation.
- **Methodology-true definitions, locked:** "MACD positive" = MACD line above zero (12 EMA > 26 EMA), evaluated at entry only, on a user-toggled 1M (default) / 5M timeframe; "rising/falling" = histogram slope vs the prior bar; VWAP/EMA distances are `(entry − level)/level × 100`. The 4-day warmup cache is more than enough for TradingView-grade MACD accuracy from the first session bar, provided you seed EMAs with the first value (adjust=false) and match session inclusion.
- **Eight build sessions, 1–2 days each:** MACD compute core → technical-state storage → backfill → tab shell + header strip → hero grid + inline-expansion framework → VWAP/EMA bands → combined reads + time-of-day → on-chart MACD toggle (closing beat). Pure TS modules + repository pattern keep it portable to the future Next.js + Postgres / SaaS / iPhone targets.

## A. Exact Metric Definitions

### A1. MACD computation (warmup-wired — bundled into Session 1)
Classic **12 / 26 / 9 on close**, confirmed as Ross's exact settings: *"What are your MACD settings? My MACD settings are standard. They are default. I do not change them."* — Fast Length 12, Slow Length 26, Signal Length 9 (Warrior Trading, "Best MACD Settings for Day Trading").

- EMA multiplier `K = 2 / (period + 1)`.
- Recursive EMA: `EMA_today = (price_today × K) + (EMA_yesterday × (1 − K))`.
- **Seed every EMA (12, 26, and the 9-period signal) with the FIRST value (adjust=false).** This matches TradingView and pandas `ewm(adjust=False)`. Do **not** use the StockCharts SMA-of-first-N-periods seed — it produces different values for the first ~50–100 bars before converging. Since FugaEdge's chart and Ross's own charts are TradingView-style, use first-value seeding for parity.
- Signal line = **9-period EMA *of the MACD line*** (not of price).
- Histogram = `MACD_line − signal_line`.

**Warmup sufficiency:** the 4-trading-day prior-day warmup cache supplies ≈1,560 one-minute bars and ≈312 five-minute bars. The slow 26-EMA seed weight decays below 0.1% after ≈90 bars, so by the very first bar of the trading session the MACD line and signal line have fully converged. The 4-day cache vastly exceeds this; **warmup length is not the parity risk.** The two real parity risks are (1) seed method (use adjust=false) and (2) session inclusion — you must match TradingView's session setting (regular-hours-only vs. including pre/post-market) when accumulating bars. Pin this to regular session unless the on-chart VWAP/MACD already use extended hours, in which case match that.

### A2. "MACD positive" — LOCKED to Ross's literal rule
Ross's rule is verbatim: *"A key point: I only trade when the MACD is positive. Trading becomes much easier when the MACD is on your side. When it's negative, I avoid taking trades, as the odds are not in my favor."* (Warrior Trading, Medium). "Positive" in his usage = **MACD line above the zero line** (12 EMA above 26 EMA), which is the front-side-of-the-move regime.

Definition: `macd_positive = (macd_line > 0)`.

He separately uses "MACD open" to mean the line is above the signal line (*"I bought the dips as long as MACD remained open, signaling continued momentum"*). These are different concepts, so **store both**:
- `macd_positive = macd_line > 0` (above zero — used for the header strip, discipline score, alignment).
- `macd_open = macd_line > signal_line` (above signal — stored for future use / secondary reads).

The four-bucket grid in Section 2 uses `sign(macd_line)` for the positive/negative axis and histogram slope for the rising/falling axis.

### A3. Rising vs falling
`rising = histogram[entry_bar] > histogram[prior_bar]` (histogram slope). This is the conventional, TradingView-aligned definition — TradingView's default MACD coloring shades bars by `hist[t] > hist[t−1]`, and histogram-slope momentum "fires earlier" than the signal-line crossover. Compare to the prior bar of the **same timeframe** selected by the 1M/5M toggle. If the entry bar is the first bar with valid history (no prior histogram), set `macd_rising = null` and exclude from the rising/falling split (still counts in positive/negative totals).

### A4. VWAP distance
Session VWAP, typical price `hlc3 = (H + L + C) / 3`, cumulative `Σ(hlc3 × volume) / Σ(volume)` from the regular-session open (09:30 ET), reset daily.

`vwap_distance_pct = (entry_price − vwap_at_entry) / vwap_at_entry × 100`

`above_vwap = vwap_distance_pct > 0`.

### A5. EMA distance
For 9 EMA and 20 EMA on the toggled timeframe:

`ema_distance_pct = (entry_price − ema_value) / ema_value × 100`

`above_9ema = ema9_distance_pct > 0`; crossover state `ema9_above_ema20 = (ema9 − ema20) > 0` (bullish stack when true).

### A6. Multi-fill entries — the evaluation point
Use **`entry_price` = volume-weighted average of all entry-side fills** for the distance numerators, but read the **indicator values (MACD/VWAP/EMA) from the bar that contains the FIRST entry fill timestamp.** Rationale: Ross's rule is an entry-decision filter, and the decision happens at the first fill; the average price reflects the position actually taken. This is a defensible, consistent convention. (Alternative considered: weight indicator state by fill — rejected as overcomplex and not how a trader reads the chart at the entry moment.)

### A7. Win rate
`win_rate = count(net P&L > 0) / total trades`. Breakeven ($0 net) counts as a **loss**. Confirmed.

### A8. Expectancy
`Expectancy = (win_rate × avg_winner) − (loss_rate × avg_loss)`

where `loss_rate = 1 − win_rate`, `avg_loss` expressed as a **positive** number, per-trade dollars. Confirmed standard (Van Tharp / TradeZella form). Example: 47 winners avg +$480, 73 losers avg −$190, 120 trades → win_rate 0.392, loss_rate 0.608 → `(0.392 × 480) − (0.608 × 190) = 188.2 − 115.5 = +$72.64` per trade. Suppress display below n=5 (show "—"); show `n=X` always so the user weighs small samples.

### A9. Discipline score
`discipline_score = % of trades where macd_positive AND above_vwap AND above_9ema`, all at entry on the toggled timeframe. The card shows the percentage plus that aligned subset's win rate and net P&L. This is the single number that operationalizes Ross's "full alignment" guidance.

## B. UI Layout (Hybrid)

**Viewport targets (Electron windowed, portable to web):** design for **min 1024px, typical 1280–1440px, max 1920px**. Below 1024px, cards stack 2-wide; below 768px, 1-wide. Use a max content width of ~1680px with centered gutters so it doesn't sprawl on ultrawide.

**Vertical order, top to bottom:**

1. **Filter bar (sticky on scroll, ~56px):** date-range presets [Today / 7D / **30D default** / 90D / YTD / Custom], ticker filter, playbook filter, and the **1M/5M segmented toggle** right-aligned. Mirrors the existing Deep Analytics filter conventions. An excluded-data chip (Section C) and an "Include paper trades" toggle (off) also live here.

2. **Header Strip (Section 1) — dashboard-card density:** a single row of **4 cards** (~120px tall) spanning full width. Cards: "MACD positive at entry", "Above VWAP at entry", "Above 9 EMA at entry", "Discipline score (full alignment)". Each: large % number, sub-line with subset win rate + net P&L. At <1280px the row wraps to 2×2.

3. **Sectioned bands (Sections 2–6) — full-width:** stacked bands, each with a section header (title + one-line plain-language question it answers) and its grid/bucket content. The hero **MACD State grid (Section 2)** sits immediately under the header strip.

**Inline expansion behavior (LOCKED interaction):** clicking any bucket/cell expands an accordion panel **directly beneath that row**, animating height ~200ms ease-out and pushing lower content (and lower sections) down. No modal, no full-tab swap; all other sections stay mounted and visible. The expanded panel contains a **trade table: 8 rows by default**, a "Show all N" expander, **sortable** columns (date, net P&L, the relevant distance/MACD value), and each row click opens the **existing trade detail modal**. Rule: only one expansion open per section at a time (opening a second bucket in the same section closes the first); different sections may be open simultaneously.

## C. Edge Cases

- **Missing indicator data** (legacy pre-feature trades, missing intraday bars, fetch failures): exclude from all technical buckets, but **never silently.** Show a persistent chip in the filter bar — "N trades excluded (no indicator data)" — with a tooltip explaining why. Header-strip percentages are computed over trades *with* data, labeled "(of N trades with data)" so the denominator is honest.
- **Sparse buckets:** always render every bucket (a stable, consistent grid is more legible than a collapsing one). Flag any bucket with **<5 trades** with a muted "low sample" badge, and **suppress expectancy** (show "—") below n=5 since it is statistical noise (reliable expectancy wants 30+ trades). Always show `n=X`.
- **Single-ticker / tiny filters:** degrade gracefully — buckets render with low-sample flags, header cards show small denominators. No hard minimum trade count; the user is trusted to read the n-counts.
- **Paper trades:** **excluded by default**, consistent with v0.2.4's separate paper-trade handling. Provide an "Include paper trades" toggle in the filter bar (off by default). Keep live and paper edges separate so the discipline read isn't polluted by sim behavior.

## D. Integration With Existing Surfaces

- **Placement — sibling tab at the top of Analytics**, beside Deep Analytics. Justification: it is the v0.2.4 centerpiece and a market-positioning feature, so it warrants top-level Analytics real estate. Rejected alternatives: a sub-tab under Deep Analytics (too buried for a flagship feature) and a top-level sidebar item (fragments the analytics grouping and breaks the mental model that "performance analysis lives in Analytics").

## E. Data Computation Strategy

**Decision: pre-compute and store per-trade indicator state in a `trade_technicals` table** keyed by `trade_id`. Do **not** compute on-the-fly at query time. Reasoning: the four-bucket grid plus Sections 3–6, recomputed on every tab open and every filter change, would require recomputing EMAs over thousands of warmup bars per trade — unacceptable latency. Pre-compute also makes the values stable, auditable, and directly syncable to future SaaS/mobile targets.

**Pure module:** `computeTradeTechnicals(trade, bars) → TradeTechnicals` (no DB/Electron imports).

**Stored columns** (entry only, per LOCKED decision — halves storage/compute), for **each timeframe (1M, 5M)**:
`macd_line, signal_line, histogram, histogram_prior, macd_positive, macd_open, macd_rising, vwap, vwap_dist_pct, ema9, ema9_dist_pct, ema20, ema20_dist_pct, ema9_above_ema20`
plus per-row: `data_complete` (bool), `computed_at`, `schema_version`.

**When it runs:**
1. **On trade import**, as the final pipeline step after the 4-day warmup bar enrichment completes (so bars are present).
2. **Lazy guard on tab open:** any trades with `data_complete = false` or a stale `schema_version` are queued for background recompute.

**Backfill strategy:** on first launch of v0.2.4 (after the Technical Analysis tab ships), run a **one-time chunked background backfill** over all historical trades (~50 trades/tick, progress toast, cancelable/resumable). Trades whose warmup bars cannot be fetched are marked `data_complete = false` and feed the excluded-data chip. After backfill, newly imported trades compute inline. Recompute-all is also triggered automatically when `schema_version` increments (e.g., a definition change).

## G. Visual Design Guidance

**MACD State grid palette** — match the already-shipped chart histogram colors, which are the lightweight-charts library defaults (confirmed in `tradingview/lightweight-charts` source `series-options.ts`: upColor default `#26a69a`, downColor default `#ef5350`):

| Bucket | Semantics | Color |
|---|---|---|
| **Positive + Rising** | Best — MACD in favor, momentum building (Ross's ideal) | strong green `#26a69a` |
| **Positive + Falling** | Caution — still positive but momentum fading | amber `#f5b041` |
| **Negative + Rising** | Recovering but still below zero — Ross says don't trade yet | muted gray-amber `#aab7b8` |
| **Negative + Falling** | Worst — Ross's explicit avoid zone | strong red `#ef5350` |

Apply the semantic color as a low-opacity (~0.12) cell-background tint; keep metric text high-contrast neutral so numbers stay readable on every tint.

**Iconography:** ▲ / ▼ for rising/falling; a small zero-line glyph for positive/negative; VWAP/EMA distance rows use a centered diverging bar (left = below the level, right = above). Flat, single-weight icons only — match the existing analytics aesthetic.

**Number formatting:**
- P&L: sign always shown, thousands separator — `+$1,250`, `−$890`; cents only below $1,000 (`−$42.50`), no cents at/above $1,000.
- Percentages: 1 decimal — `62.5%`. Distances signed — `+2.3%`, `−1.1%`.
- Expectancy: 2 decimals (per-trade) — `+$72.64`.
- Win rate: 1 decimal.
- Large numbers: abbreviate ≥$10,000 as `$12.5k` in compact cards, full value in tables and tooltips. Counts always integer.
- Divide-by-zero guard: 0 trades → "—" everywhere.

**Empty states:** zero trades after filter → centered illustration + "No trades match these filters" + "Reset filters" button. Per-bucket empty → grayed cell showing `0`, never blank. Low-sample buckets → "low sample" badge + `n=X`.

## H. On-Chart MACD Toggle (closing beat — Session 8)

- **Location:** a **4th row inside the existing "Indicators 3/3" dropdown** (label flips to "Indicators 4/4" when on). Not a separate button — keeps the chart toolbar clean and consistent with the existing indicator-management UX.
- **Persistence:** **global per-user preference** saved to settings (same store as the other indicator toggles), applied to every trade-detail modal. Not per-trade, not per-modal. Rationale: a trader either reads MACD or doesn't; per-trade state would be annoying churn.
- **Default on first launch:** **MACD pane SHOWN (on).** It is the v0.2.4 hero indicator and the basis of the whole tab; defaulting it on showcases the feature and matches Ross's actual workflow (he runs MACD at the bottom of his 1-minute chart). Fully hideable.
- **Interaction with the existing 1M/5M timeframe toggle:** the chart's timeframe toggle drives which timeframe's MACD renders; the pane recomputes from the warmup cache on switch. The **Technical Analysis tab's** 1M/5M toggle is an independent context (analytics bucketing vs. chart rendering) — they do not share state, which avoids confusing coupling between "what I'm charting" and "what I'm analyzing."

## I. Ross Cameron / DTSM Methodology — Verification & Competitive Wedge

**All definitions verified against Warrior Trading primary sources:**
- MACD rule (verbatim): *"I only trade when the MACD is positive… When it's negative, I avoid taking trades, as the odds are not in my favor."*
- Settings (verbatim): *"My MACD settings are standard. They are default. I do not change them."* (12 / 26 / 9, source = Close).
- Timeframes: 1-minute and 5-minute primary for entries; he notes the 1M becomes choppy mid-day. His prime window is the first two hours (9:30–11:30 ET), framed around a "30-minute window when MACD is most effective" with the cleanest price action — directly supporting the Section 6 time-of-day cross-cut (the buckets pre-9:30 / 9:30–10:00 / 10:00–11:00 / 11:00–12:00 / 12:00+ map onto exactly the volatility regime he describes).
- VWAP as *"the equilibrium point… If the price is above equilibrium, the bulls are in control"*; named setups: premarket break of VWAP (long) and fade-off-VWAP (short). VWAP is intraday-only, identical across intraday timeframes — supports a single VWAP-distance read regardless of the 1M/5M toggle.
- 9 EMA and 20 EMA as the standard intraday averages, with the 9/20 crossover used as a momentum filter (*"A positive crossover tells me momentum is shifting back in my favor"*) — supports Section 4's crossover state. 200 EMA daily-only for support/resistance. 10-cent stops, front-side-of-the-move focus, MACD as confirmation (not standalone).
- His scan profile (context for the trader base): stocks **$2–$20, up ≥10% on the day, RVOL ≥5×, float under 20M shares** — confirms FugaEdge's small-cap-momentum positioning and why float-bucket data already in the app is a natural future cross-cut.

**Competitor gap analysis (Tradervue, TradesViz, Tradezella, TraderSync, Edgewonk, Stonk Journal):**
- *What they do well:* **TradesViz** correlates P&L/volume/trade-count against ~10 technical indicators (ATR, ADX, RSI, MFI, CCI, TSI) across ~10–20 ranges with click-to-explore and "Apply as Filter" — the closest analog to what FugaEdge is building, and a strong UX to emulate (click a bar → filter everything). **Tradezella** does setup/time/emotion breakdowns, per-setup expectancy, and a "Compare" feature (split any two trade sets) plus best-exit/MFE-MAE. **TraderSync** does MFE/MAE (price and position). **Tradervue** offers chart studies (e.g., EMA overlays).
- *What they all MISS — FugaEdge's wedge:* none compute **per-trade indicator STATE AT ENTRY** for the specific small-cap momentum stack (MACD-positive, VWAP distance, 9/20 EMA distance and crossover). TradesViz's correlation is against **market-wide EOD indicators** (e.g., SPY's ATR), not the trader's own intraday entry alignment on the traded ticker. And none frame the analysis around a **named, community-canonical methodology** — there is no "did I follow Ross's MACD-positive rule, and does it actually pay off for me" report anywhere in the category. FugaEdge's **discipline score** and **four-bucket MACD-state-at-entry grid** are genuinely novel and are the credible basis for the "go-to journal for the DTSM community" positioning. Adopt TradesViz's click-to-filter interaction quality; differentiate on entry-state precision and methodology framing.

## J. Future-Proofing

- **Portable computation:** all indicator math lives in **pure TS modules** — `macd.ts`, `vwap.ts`, `ema.ts`, `technicals.ts` — with **zero Electron / better-sqlite3 imports**. The identical modules run server-side on the future Next.js + Postgres port.
- **Repository pattern:** `trade_technicals` access sits behind a repository interface; the better-sqlite3 implementation now and a Postgres implementation later share that interface. **Components import only from `/src/api`** — they never touch repositories or pure modules directly.
- **Internationalization (Lao's Italian translation + future locales):** every label, section title, and tooltip flows through a string table from day one — no hardcoded UI copy in components. All numbers/currency/percentages go through a single **locale-aware formatter** so Italian (and others) get correct separators and currency placement for free.
- **SaaS / iPhone companion / FugaEdge Pro:** pre-computed `trade_technicals` rows are small and serializable, so they sync directly to a mobile companion or a Pro cloud tier **without recomputation on device**. The `schema_version` column enables forward-compatible migrations and selective recompute when a definition changes. Entry-only storage (the LOCKED decision) keeps the sync payload and the cloud storage footprint minimal — a real advantage when this moves to a metered SaaS backend.
