# FugaEdge v0.2.5 — "Identity" Specification

**Bottom line:** v0.2.5 adds the identity layer — local profile,
process-based XP/levels 1–99, goals/challenges with badges, streaks —
plus a dashboard overhaul, the Edge Insights → Edge Intelligence
rebrand with an Edge Score, the sentiment polarity flip with animated
icons, a categorized Settings IA, and a signed-key access gate. All
desktop-local. Every piece is architected so the future Next.js +
Postgres SaaS inherits it (pure modules, repos, ULIDs, event-sourced
XP) instead of rewriting it. The one non-negotiable law: **XP attaches
to process, never to P&L outcome. No XP is ever lost for losing
money.** A journal must make logging the worst day the most rewarded
act, not the punished one.

Server-side features — leaderboard, trader verification, public
profiles, subscriptions — are explicitly the SaaS milestone, NOT this
release.

  non-deleted trades have playbook_id). Non-trading days (no trades,
  no no-trade mark) are NEUTRAL — skipped, never break the streak.
  Mercy: 1 streak-freeze earned per 30 consecutive journaled days,
  max 2 banked, auto-consumed on the first missed journaled-day.
  A red P&L day can never break a streak — only not journaling can.
- **D10** Edge Score v1 axes/weights/bands as specified in §F. Bands
  are informed v1 defaults with a recalibration trigger (per the

- **D11** Confidence chips on insights: Low n=5–9, Medium n=10–19,

- **D12** XP emission architecture: launch-armed reconciliation sweep
  (chunked, mirrors the technicals-backfill pattern) is the source of
  truth; inline hooks at action sites are latency optimization only.
  Hooks are safe to miss; the sweep always converges. Both paths go
  through the same pure award engine + idempotency keys.
- **D13** Idempotency key formats: `session:{date}`,
  `session_archive:{date}`, `annotate:{trade_key}`,
  `discipline:{trade_key}`, `streak:{date}`,
  `weekly_review:{week_start}`, `goal:{goal_ulid}:completed`,
  `badge:{badge_id}:{tier}`. `{trade_key}` = trades.content_hash
  when non-NULL, else `'id:' + trades.id`. Locked by the Session 1
  recon: soft-delete → re-import resurrects the same row in place
  (electron/import/repo.ts resurrectTrip — id and hash both stable),
  and hard-delete → re-import mints a new id but recomputes the
  identical content_hash, so the hash is the stable key across every
  delete → re-import of the same fill data. content_hash is unique
  among live rows (partial UNIQUE idx_trades_content_hash); the
  `id:` fallback applies only to legacy pre-v0.2.1 rows the
  migration left NULL (collision losers). Honest edge: hard-deleting
  such a NULL-hash legacy row and re-importing its fills mints a
  hash-keyed row whose annotate/discipline events can re-award once
  — and only if the original hash-owning row is also gone (otherwise
  the re-import dedups against the live owner and never lands) —
  accepted; hard delete is explicitly destructive, the population is
  the small recorded collision count, and daily caps bound the leak.
- **D14** Sidebar width locked at the shipped 180px; MASTER.md §4's
  240px is amended to match code.
- **D15** Running P&L bars are REPLACED by the cumulative equity
  curve panel. No bars/curve toggle.
- **D16** i18n readiness convention: all NEW v0.2.5 UI copy lives in
  per-feature `strings.ts` modules (plain typed objects). No i18n
  library this release; extraction later becomes mechanical.
- **D17** New dependencies (exactly two): `ulidx` (ULID generation),
  `@noble/ed25519` (activation verify; pure JS, runs in main and a
  future web backend). CC verifies license + size at install time.
  html-to-image is NOT adopted; if Phase E capture fidelity
  fails, adopting it as a third dep requires a spec amendment
  (see D21 fallback).
- **D18** xp_events are append-only and never revoked — not on trade
  soft-delete, not on goal deletion. Idempotency prevents re-award on
  restore/re-import. (Revocation creates perverse incentives and
  complexity for zero integrity gain.)
- **D19** XP law upheld absolutely for goals: equity goals award
  ZERO XP — milestones are celebration toasts only; completion
  awards the user-authored challenge badge + celebration. Process
  goals award goal_completed +1,000. Rationale: equity-target XP
  would (a) reward omitting red-day imports (cumulative P&L since
  start_date rises when losses go unlogged — the exact
  journal-corruption mode the law exists to prevent) and (b) let
  account size buy levels, breaking level-as-process-identity

  commemorate user-chosen outcomes; XP is process currency and
  may not.

- **D21** Card capture: offscreen BrowserWindow +
  `webContents.capturePage()` over a dedicated `#/card-export`

## A. XP system

### A1. Level curve
Total XP to reach level L: `totalXp(L) = 16 × (L − 1)²`. Cost of
level L: `16 × (2L − 3)` for L≥2 (arithmetic ramp, +32 per level).
Level from XP: `L = min(99, floor(1 + sqrt(xp / 16)))`.

| Level | Total XP | | Level | Total XP |
|---|---|---|---|---|
| 1 | 0 | | 60 | 55,696 |
| 5 | 256 | | 70 | 76,176 |
| 10 | 1,296 | | 80 | 99,856 |
| 20 | 5,776 | | 90 | 126,736 |
| 30 | 13,456 | | 95 | 141,376 |
| 40 | 24,336 | | 99 | 153,664 |
| 50 | 38,416 | | | |

Pacing (design target, verified): a consistent ~20-session/month
trader earns ≈2,900 XP/month from the daily loop → level ~14
after month 1, ~34 at 6 months, ~47 at 12 months, level 99 in
≈4.5 years; process-goal completions (+1,000 each) pull these
forward. Target band (mid-levels in 6–12 months, 99 in 3–5
years) holds.

### A2. Award table (the only XP sources in v0.2.5)

| event_type | XP | Cap | Notes |
|---|---|---|---|
| session_journaled | +40 | 1/date | import date within 7 days of session date (D4) |
| session_journaled_archive | +10 | 1/date, cap-exempt | older than 7 days (D4) |
| trade_fully_annotated | +12 | 6/day | per D8 definition |
| disciplined_entry | +15 | 4/day | per D7 predicate |
| daily_streak_bonus | +25 | 1/date | per D9 journaled-day |
| weekly_review_completed | +175 | 1/week | per D5 button |
| goal_completed | +1,000 | per process goal | process goals only (D19) |

The daily-capped types sum to a structural max of 197 XP/day — the
~200/day soft cap is emergent, not a separate mechanism. Weekly
review, process-goal completions, and archive sit outside it. NO
event type may ever reference P&L sign, trade count, position
size, or dollar/equity targets — equity-goal progress is
celebrated and badged, never XP'd (D19).

### A3. Storage & engine
- `xp_events` (append-only): `id TEXT PRIMARY KEY` (ULID),
  `event_type TEXT NOT NULL`, `source_ref TEXT`, `xp INTEGER NOT
  NULL`, `idempotency_key TEXT NOT NULL UNIQUE`, `created_at TEXT
  NOT NULL` (ISO UTC). Total XP = SUM(xp); level derived. No mutable
  current-XP row anywhere.
- Pure modules (zero Electron/DB imports): `src/core/xp/curve.ts`,
  `src/core/xp/awards.ts` (eligibility predicates over plain data),
  `src/core/xp/engine.ts` (given facts → list of award intents with
  idempotency keys). Repos: `electron/xp/{repo,ipc}.ts` +
  `shared/xp-types.ts` per the session/technicals precedent.
- Reconciliation sweep (D12): launch-armed after the existing warmup
  → technicals chain; enumerates facts (sessions by date, annotated
  trades, discipline-eligible technicals rows, journaled days, weeks
  with completion events, goal states), computes intents via the pure
  engine, INSERT OR IGNORE on idempotency_key. Chunked ~200
  intents/tick, setImmediate-yielded, progress channel
  `XP_RECONCILE_PROGRESS`. Second consecutive run on an unchanged DB
  MUST produce zero new events (the core idempotency invariant).
- Inline hooks (latency only): after import `commit()` returns
  (electron/import/ipc.ts, insertedTrips known), after the four
  annotation save channels (TRADE_PLAYBOOK_SAVE, TRADE_CATALYST_SAVE,
  TRADE_NOTE_SAVE, TRADE_MISTAKES_SAVE — re-evaluate D8 on each),
  on the weekly-review button, on goal evaluation. Each hook calls
  the same engine; misses are healed by the sweep.

## B. Schema & identity tables

All new tables via CREATE TABLE IF NOT EXISTS in SCHEMA_SQL (the
trade_technicals precedent — lands on fresh installs and upgrades
with no migration module and NO version bump; L1, shipped in Phase A
Session 1). The SCHEMA_VERSION 28 → 29 bump ships in Phase C
alongside the sentiment remap migration, whose settings latch it
arms. The XP reconciliation sweep needs no arming — it is idempotent
and self-gating on every launch (D12).

- `profile` (single row): `id TEXT PRIMARY KEY` (ULID),
  `display_name TEXT`, `handle TEXT`, `avatar_data TEXT`,
  `trading_style TEXT`, `markets TEXT`, `bio TEXT`,
  `featured_badges_json TEXT NOT NULL DEFAULT '[]'` (≤3 ids),
  `member_since TEXT`, `created_at TEXT`, `updated_at TEXT`.
  Avatar stored as a data-URL (`avatar_data`), downscaled to
  ≤256px renderer-side via canvas before save — no
  attachments-service extension; travels with DB backups and
  future sync (D20). Onboarding's currently-discarded
  TradingStyle answer now persists here.
- `goals`: `id TEXT PRIMARY KEY` (ULID), `title TEXT NOT NULL`,
  `kind TEXT NOT NULL` ('equity'|'process'), `config_json TEXT NOT
  NULL` (equity: start_date, start_amount, target_amount; process:
  metric, target, window), `status TEXT NOT NULL`
  ('active'|'completed'|'abandoned'), `created_at TEXT`,
  `completed_at TEXT`.
- `badge_awards`: `id TEXT PRIMARY KEY` (ULID), `badge_id TEXT NOT
  NULL`, `tier TEXT` ('copper'|'silver'|'gold'|NULL),
  `awarded_at TEXT NOT NULL`, `source_ref TEXT`. Dedup is the
  expression index `idx_badge_awards_identity` on
  `(badge_id, IFNULL(tier, ''))` — a table-level UNIQUE(badge_id,
  tier) would NOT dedupe untiered (NULL-tier) awards because SQLite
  treats NULLs as distinct in unique constraints, and challenge
  badges are exactly that. Badge CATALOG is code, not data:
  `src/core/badges/catalog.ts`.
- Sentiment polarity migration `migrate-sentiment-polarity.ts`:
  version-gated + settings latch + throwing pre-migration backup per
  the established pattern; one-shot
  `UPDATE session_meta SET sentiment = 6 - sentiment WHERE sentiment
  IS NOT NULL`. The full 10-point touch list from the 2026-06 recon
  applies verbatim (SENTIMENT_LABELS reorder; the duplicate label
  record at electron/analytics/get.ts:738-745; the four
  polarity-encoded color maps — SentimentSelector.toneFor,
  CalendarGrid.SentimentBadge, TodaySessionCard, 
  SentimentBreakdownCard.LevelDot; runSentimentEdge thresholds s<=2 /
  s>=4 flip to s>=4 / s<=2 + body copy; Calendar.tsx cycle comments;
  copy strings incl. today.ts:158 'Sentiment too cold' and tour
  steps; schema.ts comment; sentiment test fixtures). After the
  flip: 5='3+ stocks >100%' (best) … 1='0 stocks >50%' (worst).
  Historical views prefer the ICON over the raw number.

## F. Edge Intelligence + Edge Score

- Rename all "Edge Insights" surfaces → "Edge Intelligence". New nav

- Tab content: full insight feed (every rule, tone borders kept),
  per-card n-count ALWAYS visible, D11 confidence chips, n<5

  Edge Score detail (radar + published weights + per-axis raw values

- Dashboard band: three cards — Biggest Edge (highest-priority

  (action derived from the leak) — a pure selector over
  runAllInsightRules output. No new detection engine.
- Edge Score (pure `src/core/score/edgeScore.ts`), 0–100, six axes:

| Axis | Weight | v1 band (score 0 → 100) |
|---|---|---|
| Discipline (full-alignment % from technicals) | 25% | 0% → 100% direct |
| Profit Factor | 20% | ≤0.8 → ≥2.5 linear |
| Win Rate | 15% | ≤30% → ≥65% linear |
| Avg Win/Loss | 15% | ≤0.5 → ≥2.0 linear |
| Max Drawdown / gross profit (inverted) | 15% | ≥1.0 → ≤0.2 linear |
| Consistency (% green days) | 10% | ≤30% → ≥70% linear |

  Discipline is deliberately the heaviest axis — the score can rise
  in a flat-P&L month if process improves. Bands are v1 defaults;
  recalibration trigger: ~200–500 trades per user (v0.2.4 §A4
  precedent). n<5 trades in window → score suppressed; n<20 →
  "provisional" chip. Weights and bands are rendered in-app (the
  published-formula rule).

## Acceptance criteria

**Phase A:** re-importing the same file produces ZERO new xp_events.
Historical import (dates >7 days old) produces archive events only,
cap-exempt. Daily-capped types cannot exceed 197 XP/day. Running the
reconciliation sweep twice consecutively on an unchanged DB produces
zero new events on the second run. Level derivation tested at every
A1 milestone. xp_events never revoked on trade soft-delete (D18
test). v0.3.0 ideas doc gains the XP-sites paper-audit line.
**Phase B:** process-goal completion emits exactly one
goal_completed event + badge; equity-goal completion and equity
milestones emit ZERO xp_events (badge + celebration only);
featured badges hard-capped at 3; deleting a goal revokes nothing.
**Phase C:** sentiment migration run against a COPY of the real dev
DB flips every display coherently (labels, four color maps, insights
copy, analytics labels — spot-check list from §B); reduced-motion
shows zero animation; the equity curve's values match the old
Running P&L tooltip cumulative exactly; zero-trade dashboard renders
the intentional empty state.
**Phase D:** every insight card shows n; nothing renders below n=5;
score suppressed n<5 and provisional n<20; weights/bands visible
in-app.
**Phase E:** an existing-data DB is NEVER locked out (grace →
read-only → export verified); exported card is crisp at 2× with
correctly rendered JetBrains Mono and contains no dollars by
default; sandbox outcome documented either way; all new copy in
strings modules.
