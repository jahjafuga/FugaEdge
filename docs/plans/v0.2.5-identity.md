# FugaEdge v0.2.5 — "Identity" Specification

**Bottom line:** v0.2.5 adds the identity layer — local profile,
process-based XP/levels 1–99, goals/challenges with badges, streaks —
plus a dashboard overhaul, the Edge Insights → Edge Intelligence
rebrand with an Edge Score, the sentiment polarity flip with animated
icons, a categorized Settings IA, and a signed-key access gate. All
desktop-local. Every piece is architected so the future Next.js +
Postgres SaaS inherits it (pure modules, repos, ULIDs, event-sourced
XP) instead of rewriting it. The one non-negotiable law: **XP attaches
to process, never to P&L outcome. No XP is ever lost for losing
money.** A journal must make logging the worst day the most rewarded
act, not the punished one.

Server-side features — leaderboard, trader verification, public
profiles, subscriptions — are explicitly the SaaS milestone, NOT this
release.

- **D22** XP engine mechanics (Session 2 rulings L4–L9, condensed).
  source_ref conventions: per-trade events carry the TRADE'S DATE
  (YYYY-MM-DD); session/streak events the date; weekly_review the
  week_start; goal_completed the goal ULID — per-date cap accounting
  is computable from the ledger alone and survives hard-deleted
  trades without joins. Caps are per trade-date with TOP-UP:
  remaining = cap − existing same-type events whose source_ref is
  the date (clamped ≥0); candidates are taken by trades.id
  ascending; an existing event whose trade was hard-deleted still
  consumes its slot via source_ref; a null/legacy source_ref
  consumes none (defensive — the convention ships from day one).
  The engine CONVERGES with whatever the inline hooks already paid:
  if the cap is exhausted it emits zero even where id-order would
  have picked differently. Fresh window (the D4 amendment):
  per-trade XP (annotation, discipline) is awardable only while the
  trade's date is ≤ 7 UTC days before now — day-7 inclusive, day-8
  exclusive; older trades never earn per-trade XP. This kills both
  the pre-annotated-history XP windfall and the wall-clock dribble
  while preserving D4's head-start intent (history is rewarded via
  archive session XP + the Historian badges). Accepted edge:
  technicals that compute >7 days after the trade date forfeit the
  discipline bonus — rare post-§K, under-pay direction. Session
  fresh/archive: session_journaled (+40) iff importedAt is non-null
  AND floor-days(importedAt − sessionDate) ≤ 7; otherwise archive
  (+10), INCLUDING importedAt = null. importedAt is derived from
  trades.created_at (MIN over the date's non-deleted trades, so a
  straggler fill can never re-qualify an old session as fresh),
  format-tolerant per the date-prefix rule (the engine consumes
  only the YYYY-MM-DD prefix, component-parsed as UTC); null
  remains the defensive fallback only. A date is eligible only if
  NEITHER session:{date} NOR session_archive:{date} exists — one
  logical award at two rates, never both. Session vs streak bars:
  the session award requires ≥1 non-deleted trade on the date (it
  rewards the IMPORT act); daily_streak_bonus requires the D9
  journaled-day predicate; a no-trade day earns the streak bonus
  only; a zero-trade day with no no-trade mark is neutral and earns
  nothing. Engine properties, both tested: DETERMINISM — identical
  input → identical intents including order (sorted date asc, then
  event_type, then trade id); SUBSET-SAFETY — the engine over any
  subset of dates emits exactly that subset's slice of the
  full-input output, which is what lets the inline hooks and the
  reconciliation sweep share one function (D12). One deliberate
  asymmetry, recorded so no future session mistakes it for a bug:
  daily_streak_bonus has NO fresh window. It retro-credits any date
  meeting the D9 bar — a bar a CSV import alone can never satisfy;
  only per-day journaling work in the app can (sentiment + 100%
  tagging, or a no-trade mark). As a farming vector it is worthless
  by construction: fully tagging an entire old day pays 25 XP while
  one fresh annotated trade pays 12 — archive-tier pay for real
  process work. First-launch consequence: a diligently journaled
  history retro-pays its streak bonuses (cohort reality ≈ a
  level-10 head start), which is the intended reward for journaling
  actually performed, not a windfall.
- **D23** XP emission layer (Session 3 rulings L10–L16, condensed).
  Sweep shape (corrects the §A3 forecast): runXpReconcile is
  SINGLE-PASS — no chunking, no progress channel. §K chunked because
  its items were rate-limited network fetches; this is local SQL +
  pure math, sub-second at 5,000-trade scale. It mirrors the §K
  siblings in WIRING only: launch-armed third in the awaited
  warmup → technicals → xp chain (order load-bearing — technicals
  flips stubs complete, THEN xp awards them), try/catch log-only,
  fast no-op at steady state, one '[FE launch backfill] xp:' summary
  line; the same third link rides the Settings-refresh chain. Hook
  set (corrects spec §A3): hooks attach to the channels whose writes
  can change award eligibility — IMPORT_COMMIT, TRADE_PLAYBOOK_SAVE
  (D8+D9), TRADE_CATALYST_SAVE (D8), TRADE_NOTE_SAVE (D8),
  SESSION_SENTIMENT_SAVE (D9), SESSION_TODAY_SAVE (D9);
  TRADE_MISTAKES_SAVE and TRADE_CONFIDENCE_SAVE are NOT hooked —
  neither feeds D7/D8/D9, so a reconcile there is a pointless no-op
  per save. Hook semantics: fire-and-forget, appended AFTER save +
  bumpDataVersion; a hook failure delays XP by one launch (the sweep
  heals), never loses it, never breaks a save, never changes handler
  latency or return shape. The Promise.resolve().then() deferral in
  every hook is LOAD-BEARING, not cosmetic — the reconcile functions
  are synchronous, so an inline call would execute inside the
  handler and add its latency to every save, violating this very
  guarantee; do not "simplify" it away. The import hook derives its
  date set from ALL trips in the commit input (inserted +
  resurrected + duplicate) — over-reconciling a paid date is free by
  idempotency. Fresh-
  import discipline timing (emergent and correct — do NOT "fix" it
  with import-path special-casing): a just-committed import has no
  technicals rows yet, so the import hook mints session/streak
  events only; the discipline bonus for those trades lands via the
  NEXT reconcile that sees technicals — in the realistic flow the
  user's own annotation-save hook (reconcileXpForDates re-assembles
  the whole date), else the refresh chain or next launch. Assembly
  scope: TradeFact assembly is restricted to the fresh window —
  provably identical engine output via L6 + the L9 subset-safety
  property; SessionFact assembly covers the FULL history (universe =
  dates with ≥1 non-deleted trade UNION session_meta dates, so
  no-trade journaled days are included); existing-event assembly
  reads the full ledger projection (fine at this scale). The XP
  assembly layer is structurally P&L-BLIND by construction — no
  facts query selects any P&L column, enforced by a guard test —
  defense-in-depth for the §A2 law. importedAt = per-date
  MIN(trades.created_at) over non-deleted rows, passed RAW (the
  engine consumes the date prefix). No dataVersion bump on XP writes
  — there is no renderer surface yet; REVISIT the moment Phase B
  ships XP UI (hooks will then need renderer reactivity). Weekly
  review: the two channels ship in Session 3

  { completed: false, error }; 'xp:weeklyReviewGet' — idempotency-

  Session 6 (D5's button now has an owner).

## A. XP system

### A1. Level curve
Total XP to reach level L: `totalXp(L) = 16 × (L − 1)²`. Cost of
level L: `16 × (2L − 3)` for L≥2 (arithmetic ramp, +32 per level).
Level from XP: `L = min(99, floor(1 + sqrt(xp / 16)))`.

| Level | Total XP | | Level | Total XP |
|---|---|---|---|---|
| 1 | 0 | | 60 | 55,696 |
| 5 | 256 | | 70 | 76,176 |
| 10 | 1,296 | | 80 | 99,856 |
| 20 | 5,776 | | 90 | 126,736 |
| 30 | 13,456 | | 95 | 141,376 |
| 40 | 24,336 | | 99 | 153,664 |
| 50 | 38,416 | | | |

Pacing (design target, verified): a consistent ~20-session/month
trader earns ≈2,900 XP/month from the daily loop → level ~14
after month 1, ~34 at 6 months, ~47 at 12 months, level 99 in
≈4.5 years; process-goal completions (+1,000 each) pull these
forward. Target band (mid-levels in 6–12 months, 99 in 3–5
years) holds.

### A2. Award table (the only XP sources in v0.2.5)

| event_type | XP | Cap | Notes |
|---|---|---|---|
| session_journaled | +40 | 1/date | import date within 7 days of session date (D4) |
| session_journaled_archive | +10 | 1/date, cap-exempt | older than 7 days (D4) |
| trade_fully_annotated | +12 | 6/day | per D8 definition; fresh window + per-date cap (D22) |
| disciplined_entry | +15 | 4/day | per D7 predicate; fresh window + per-date cap (D22) |
| daily_streak_bonus | +25 | 1/date | per D9 journaled-day |
| weekly_review_completed | +175 | 1/week | per D5 button |
| goal_completed | +1,000 | per process goal | process goals only (D19) |

The daily-capped types sum to a structural max of 197 XP/day — the
~200/day soft cap is emergent, not a separate mechanism. Weekly
review, process-goal completions, and archive sit outside it. NO
event type may ever reference P&L sign, trade count, position
size, or dollar/equity targets — equity-goal progress is
celebrated and badged, never XP'd (D19).

### A3. Storage & engine
- `xp_events` (append-only): `id TEXT PRIMARY KEY` (ULID),
  `event_type TEXT NOT NULL`, `source_ref TEXT`, `xp INTEGER NOT
  NULL`, `idempotency_key TEXT NOT NULL UNIQUE`, `created_at TEXT
  NOT NULL` (ISO UTC). Total XP = SUM(xp); level derived. No mutable
  current-XP row anywhere.
- Pure modules (zero Electron/DB imports): `src/core/xp/curve.ts`,
  `src/core/xp/awards.ts` (eligibility predicates over plain data),
  `src/core/xp/engine.ts` (given facts → list of award intents with
  idempotency keys). Repos: `electron/xp/{repo,ipc}.ts` +
  `shared/xp-types.ts` per the session/technicals precedent.
- Reconciliation sweep (D12): single-pass `runXpReconcile` — no
  chunking, no progress channel — launch-armed as the third link of
  the warmup → technicals → xp chain (rationale and full mechanics:
  D23). Second consecutive run on an unchanged DB MUST produce zero
  new events (the core idempotency invariant — proven in the
  Session 3 real-journal fixture smoke).
- Inline hooks (latency only; hook set per L11/D23): hooks attach to
  the channels whose writes can change award eligibility —
  IMPORT_COMMIT (session/streak), TRADE_PLAYBOOK_SAVE (D8 + D9),
  TRADE_CATALYST_SAVE (D8), TRADE_NOTE_SAVE (D8),
  SESSION_SENTIMENT_SAVE (D9), SESSION_TODAY_SAVE (D9).
  TRADE_MISTAKES_SAVE and TRADE_CONFIDENCE_SAVE are NOT hooked —
  neither feeds D7/D8/D9, so a reconcile there is a pointless no-op
  per save. The weekly-review button and goal completion emit their
  intents directly (constructors, not reconcile hooks). Each hook
  calls the same engine; misses are healed by the sweep.

## F. Edge Intelligence + Edge Score

### Design north-star (founder reference mockup, 2026-06-13)

CORE CONCEPT ("the weapon"): Edge Intelligence is PRESCRIPTIVE, not
descriptive — it interprets the trader's OWN data and tells them what
to do. The spine is three cards: **Biggest Edge** (where profit
concentrates — e.g. "82% of profits before 10AM, +$X contribution"),
**Biggest Leak** (where P&L bleeds — e.g. "Cold Market Days, −$X,
win-rate drop A%→B%"), and **Focus Area** (the prescription WITH a
dollar value — e.g. "Reduce size after 11AM, +$X/mo est."). Plus an
**Edge Score** (single 0–100 "how sharp am I" + a vs-last-30-days
delta), **Trader DNA** (archetype + confidence % + best / weak
conditions), and a **Trading Coach** that renders the findings as
plain-English directives. Output voice = the leak, the edge, and the
one dollar-quantified change worth making.

VISUAL TARGET (Phase E, NOT the build phase): a futuristic aesthetic —
violet / teal glow, a DNA / fingerprint motif for Trader DNA, gradient
depth on dark. Edge SHIPS in its build phase with clean D26-grammar UI
(structure, cards, data, prescriptive copy all REAL); the futuristic
realization lands in the coordinated Phase E design sweep — never a
polished one-off in the engine phase.

SCOPE LINE: the reference mockup also shows dashboard-only elements
(equity curve, bottom KPI strip, Net P&L hero) — those belong to the
CAPSTONE DASHBOARD, not Edge Intelligence. Edge builds its engine + its
own surface; the dashboard borrows Edge's widgets LAST (per the
build-order resequence above).

PROCESS-NOT-P&L BOUNDARY: Edge Intelligence ANALYZES P&L — it is an
analytics surface, and that is legitimate. But it must NEVER feed the
XP / badge reward system, which stays process-only (D19). Edge reads
P&L to ADVISE; the reward system reads the ledger to RECOGNIZE
behavior — the two never cross.

- Rename all "Edge Insights" surfaces → "Edge Intelligence". New nav
  item `/intelligence` (main list, after Analytics).
- Tab content: full insight feed (every rule, tone borders kept),
  per-card n-count ALWAYS visible, D11 confidence chips, n<5
  suppressed; session/week summary ("What worked / What leaked"
  two-column, generated from rule outputs for the selected range);
  Edge Score detail (radar + published weights + per-axis raw values
  on hover).
- Dashboard band: three cards — Biggest Edge (highest-priority
  positive-tone insight), Biggest Leak (highest negative), Focus Area
  (action derived from the leak) — a pure selector over
  runAllInsightRules output. No new detection engine.
- Edge Score (pure `src/core/score/edgeScore.ts`), 0–100, six axes:

| Axis | Weight | v1 band (score 0 → 100) |
|---|---|---|
| Discipline (full-alignment % from technicals) | 25% | 0% → 100% direct |
| Profit Factor | 20% | ≤0.8 → ≥2.5 linear |
| Win Rate | 15% | ≤30% → ≥65% linear |
| Avg Win/Loss | 15% | ≤0.5 → ≥2.0 linear |
| Max Drawdown / gross profit (inverted) | 15% | ≥1.0 → ≤0.2 linear |
| Consistency (% green days) | 10% | ≤30% → ≥70% linear |

  Discipline is deliberately the heaviest axis — the score can rise
  in a flat-P&L month if process improves. Bands are v1 defaults;
  recalibration trigger: ~200–500 trades per user (v0.2.4 §A4
  precedent). n<5 trades in window → score suppressed; n<20 →
  "provisional" chip. Weights and bands are rendered in-app (the
  published-formula rule).
