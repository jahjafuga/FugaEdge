\## Why this release exists



Within 24 hours of v0.1.6 going public on GitHub, three real momentum traders found FugaEdge and tried it. All three hit walls at the import step:



\- Tester A (DTSM, DAS Trader) — tried execution-level CSV, blocked

\- Tester B Hogan (DTSM founder, DAS Trader) — same execution CSV, blocked

\- Webull user (DTSM connection unknown) — tried both Webull mobile CSV and desktop XLSX, both blocked



v0.1.6 only parsed one CSV format: the DAS Trader summary export, which is what Lao personally uses. Most experienced DAS users export execution-level data instead. Webull users export entirely different formats.



v0.2.0 fixes this by introducing a universal import architecture: one internal data model, multiple parsers per broker/format, auto-detection on import.



This is the release that unblocks the DTSM beta cohort and opens FugaEdge to the Webull user base.

\## Architecture



\### Universal data model



Two internal types, broker-agnostic:



\*\*Execution\*\* (one fill):

\- id, source\_broker, source\_format, source\_file

\- timestamp\_utc, symbol, side, quantity, price

\- order\_id (for grouping partial fills)

\- route, liquidity\_type

\- commission, ecn\_fee, sec\_fee, finra\_fee, cat\_fee, htb\_fee, other\_fees

\- account\_name, is\_paper



\*\*RoundTrip\*\* (aggregated from executions):

\- id, symbol, account\_name, side (LONG/SHORT)

\- entry\_time\_utc, exit\_time\_utc

\- shares, avg\_entry\_price, avg\_exit\_price

\- gross\_pnl, total\_fees, net\_pnl, fees\_reported (boolean)

\- execution\_ids (links back)

\- User fields: playbook, catalyst\_type, mistake, confidence, notes

\- Enriched: float, country, market\_cap



\### Adapter pattern



Each broker/format gets a thin parser that produces Execution\[]:



\- DAS Summary CSV → adapter → Execution\[]

\- DAS Execution CSV → adapter → Execution\[]

\- DAS Account Report CSV → adapter → fee data matched to existing executions

\- Webull Mobile CSV → adapter → Execution\[]

\- Webull Desktop XLSX → adapter → Execution\[]



After parsing, the rest of FugaEdge doesn't care about source broker.



\### Round trip builder



One function, broker-agnostic:



Takes Execution\[] sorted by timestamp.

For each symbol:

\- Tracks running position

\- Detects when position goes from 0 to non-zero (new round trip)

\- Detects when position returns to 0 (round trip ends)

\- Handles adds to position (weighted avg entry)

\- Handles short selling (Sell before Buy = short)

\- Handles partial fills via order\_id grouping

\- Handles open positions at end of file



Returns RoundTrip\[].



\### Import pipeline



1\. User drops file(s)

2\. Detect format(s) from header sniffing

3\. Parse each file → Execution\[] or fee data

4\. Merge executions, deduplicate

5\. Build round trips via universal builder

6\. Match fees if Account Report present

7\. Enrich async (float, country)

8\. Save to database

9\. Show result with progress UI

\## Architecture decisions



\### Decision 1 — Date handling for execution CSVs (no date column)



DAS execution CSVs only have time, not date. Resolution:



\- If filename matches YYYY-MM-DD.csv or MM-DD-YYYY.csv → use that

\- Otherwise → show date picker, prompt user

\- If user also drops Account Report → use that file's date and skip prompt



\### Decision 2 — Multi-file import UX



User can drop one or both files at once:



\- If both detected → process combined

\- If execution only → process but show banner: "Add Account Report CSV for fee data"

\- If summary only → process as before (backwards compatible)



\### Decision 3 — Migration for existing v0.1.6 data



Full one-time migration on first v0.2.0 launch:



\- Auto-backup database before migration (rollback option)

\- Show progress banner: "Upgrading database for v0.2.0..."

\- Convert all existing trades to new universal model

\- Verify integrity with checksum

\- Tests verify nothing is lost

\### Day 7.5 — Database safety patch

Two-piece safety net. Auto-backup is the entire recovery story in v0.2.0 since deletion / restore UI is deferred to v0.2.1. Closes the gap that bit Tester B on 2026-05-19 (wrong daily-summary CSV imported, no rollback path, manual rename-via-File-Explorer was the only recovery).

- Silent auto-backup before every import:
  - Before any import handler writes to the DB, copy fugaedge.db to %APPDATA%\fugaedge\backups\fugaedge-pre-import-<ISO-timestamp>.db
  - Applies to every parser path: DAS Summary, DAS Execution, DAS Account Report, Webull Mobile, Webull Desktop XLSX, any future format
  - Retention: keep last 20 backups, prune oldest on each new backup
  - Architecture (per ARCHITECTURE.md): file copy goes through a /src/platform/* storage abstraction so a web port can swap to server-side blob storage; trigger logic lives in /src/core or /src/lib and accepts the storage interface as a parameter
  - If backup write fails, abort the import with a clear error — do NOT import without a backup, that defeats the purpose
  - Backups folder auto-created on first import
  - Log backup path on success so DM walkthroughs reference the real file

- "Reset journal" button in Settings:
  - Single button, double-confirmation modal ("type DELETE to confirm")
  - On confirm: rename current fugaedge.db → fugaedge-reset-<ISO-timestamp>.db and restart app (fresh DB created on next launch). Same rename-not-delete pattern as auto-backup
  - No "restore from backup" UI in v0.2.0 — manual recovery only (close app, open backups folder, rename desired .db back to fugaedge.db, relaunch). Restore UI lands in v0.2.1

- Acceptance criteria:
  - Every import path produces a timestamped backup BEFORE any executions or round_trips are written to the DB
  - Backup count never exceeds 20 in the backups folder
  - Backup file is binary-identical to source (validate size + checksum)
  - Reset button rename + restart leaves a fresh empty DB and a safety-net .db file in %APPDATA%\fugaedge\
  - Second Reset click succeeds without filename collision (ISO timestamps prevent collisions in practice)

Estimated effort: 2-3 hours. Slots after Day 7 (enrichment pipeline stabilises the import-handler surface) and before Day 8 (aggregator fix + light/dark audit) so safety lands before polish.
