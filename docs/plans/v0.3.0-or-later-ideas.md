\### Trade Behavior Analytics — MFE / Reversal Tracking

Source: Mark Douglas, "Trading in the Zone", surfaced 2026-05-17



Concept: 90% of losing trades go in the trader's direction first before reversing. FugaEdge should track Maximum Favorable Excursion (MFE) and time-to-peak for every trade.



Why this matters:

\- Reveals "winners turned losers" pattern across many trades

\- Helps trader understand: setup wasn't wrong, exit was wrong

\- Drives behavior change toward scaling out

\- Mark Douglas attribution adds psychological credibility



Feature scope:

\- For each trade: compute MFE via Polygon historical aggregates

\- Show time-to-peak (how long position was at max profit)

\- Identify "winners turned losers" pattern in Edge Insights

\- Per-playbook MFE statistics

\- Optional: "scale-out simulator" showing what-if you'd taken partial profits at X% gains

\- Educational tooltip with Mark Douglas attribution



Data requirements:

\- Polygon aggregates endpoint (1-minute or 1-second resolution)

\- 1 API call per trade for backfill

\- Async enrichment queue (similar to country/float)



Why not v0.2.0:

\- v0.2.0 focus is broker support + import polish

\- Needs Polygon enrichment to be stable first

\- Deserves dedicated release (not Day 8 add-on)

\- Real depth requires proper UI work



Possible home: v0.4.0 or v0.5.0 — "Trade Behavior" theme release

\### Auto-isolate dev DB from packaged DB (testing infrastructure)

Source: three distinct failure modes hit in one evening (2026-05-16) around the dev/packaged-build split sharing `%APPDATA%\fugaedge\`. Each one cost real diagnosis time; the auto-isolation work closes all three.



Motivating incidents:

1\. **Test data leaks dev → real dashboard.** Day 2 smoke-test imports (Tester A's CSV scenarios 1-3+6, Tester B's CSVs scenarios 4-5) ran via `npm run dev` against the shared userData path, polluting the real journal dashboard. Required a manual `.bak` restore from `backups/` to recover.

2\. **Packaged shadows dev (binary-vs-DB-version mismatch).** After the restore, the next launch was the installed packaged build (`%LOCALAPPDATA%\Programs\FugaEdge\FugaEdge.exe`, built 2026-05-14 with `SCHEMA_VERSION='17'`) instead of the dev branch (`SCHEMA_VERSION='18'`). Old binary opened the restored v17 DB, saw "current = target", and silently bypassed the v17→v18 migration. No warning that two binaries with different schema targets were writing to the same file. ~30 min misdiagnosed as a schema-code bug before the file timestamp on the .exe gave it away.

3\. **Env-var bleed across dev tooling.** Running schema diagnostics under Electron's ABI required `ELECTRON_RUN_AS_NODE=1` (the better-sqlite3 binary is compiled against Electron 33's ABI via the `rebuild:sqlite` postinstall, not against system Node). The env var leaked from the manual one-liner into a follow-up `npm run dev`, causing Electron to launch in Node mode and `require('electron')` to return the path string instead of the API namespace — opaque TypeError on `protocol.registerSchemesAsPrivileged`. Addressed by `npm run inspect:schema` (commit fef5b20, cross-env-scoped), but the env-var-bleed-via-shared-shell class is the same dev/test isolation theme.



Concept: In dev mode (`!app.isPackaged`), use `%APPDATA%\fugaedge-dev\fugaedge.db` instead of `%APPDATA%\fugaedge\fugaedge.db`. Zero-friction separation between dev/test work and real journal data — `npm run dev` automatically writes to a separate DB without anyone remembering anything. Directly addresses #1 and #2 above.



Belt-and-suspenders: add a `FUGAEDGE_DB_PATH` env var override so multiple fixture sets can each have their own DB when needed (e.g., `FUGAEDGE_DB_PATH=...\tester-b-fixtures.db npm run dev`). Also closes future "I need to test against Tester A's data without touching dev state" scenarios.



Related: any dev script that needs the Electron ABI should be wrapped in a `cross-env`-scoped npm script (precedent: `inspect:schema` at fef5b20) so `ELECTRON_RUN_AS_NODE` never leaks into the user's shell session.



Trade-off to think through before implementing: auto-isolation immediately splits the current real-data dashboard from new dev runs. Anyone who currently uses `npm run dev` against real data (e.g., journaling between dev sessions) loses that view unless they launch the packaged installer. If that's acceptable, ship the auto-isolation. If not, ship env-var-override alone first and add auto-isolation later.



Possible home: v0.3.0 (small infrastructure win, frees future smoke testing from data-pollution risk; three concrete failure modes already paid the diagnosis tax).

\### Paper-account trade import + filtering

Source: Day 4-5 design decision (2026-05-16). The `executions.is_paper` storage column has existed since v0.2.0 Day 1 (commit 3f1df7d), but a grep of the codebase confirms **zero callers filter on it**. None of the dashboard, edge insights, discipline streak, expectancy math, P&L aggregations, or analytics paths know the column exists. Shipping a paper-account import UI in v0.2.0 without the filtering would store correct metadata while letting paper trades silently pollute real-account stats — the exact corruption mode v0.2.0 was supposed to prevent.



v0.2.0 mitigation (already shipped on Track C): the import preview has an "Account type" toggle defaulting to "Real account". Selecting "Paper / simulated" disables the Import button with the message *"Paper-account imports arrive in v0.3.0 once we can keep them out of your real-account stats."* Cuts the one Webull paper user from importing in v0.2.0; protects every other user from self-corruption.



v0.3.0 solution, three parts:

1\. **Enable paper-account imports** — remove the disabled-on-paper guard so the toggle actually lets the import proceed. The import code path is unchanged; only the gate is.

2\. **Audit every aggregation site** and add `WHERE is_paper = 0` (SQL) or the equivalent renderer-side filter on hydrated execution rows. Known surface: `electron/stats/dashboard.ts`, `electron/analytics/get.ts`, `electron/calendar/get.ts`, `electron/calendar/weekly.ts`, `electron/reports/get.ts`, `src/core/insights/rules.ts`. Spot-check every list / chart / metric for paper-leak after the audit.

3\. **Optional later (v0.4.0+)**: a "Show paper trades in stats" preferences toggle for users who actively want paper mixed in (rare, but a legitimate use case for backtesting cross-checks).



Estimated effort: 1-2 days for parts 1 + 2. Part 3 is a small follow-up if requested.



Blocks (in v0.2.0): the one Webull paper user in the beta cohort can't import their paper data until this ships. Communicate clearly in release notes; tooltip already does the same job in-app.



Possible home: v0.3.0 (paired with [[auto-isolate-dev-db]] under the broader "v0.2.0 left these things half-done on purpose" theme).

\### Per-date DST verification on Webull Mobile timestamps

Source: Day 4 design decision (2026-05-16). The single Webull Mobile fixture covered 2026-04-16 → 2026-05-14 — entirely inside US Daylight Saving Time, so the literal `EDT` suffix in every timestamp was seasonally correct (`UTC-04:00`). Whether Webull emits seasonally-correct abbreviations year-round (i.e. switches to `EST` after the November DST transition) is unverified — we have no winter fixture to confirm.



v0.2.0 parser behavior (already shipped on Track A):

\- Read the literal abbreviation (`EDT` or `EST`) from the timestamp.

\- Convert to UTC accordingly (`-04:00` for EDT, `-05:00` for EST).

\- **No per-date DST verification.** Trust the literal.



v0.3.0 enhancement: for each parsed Webull Mobile timestamp, compute the actual DST status for that date in `America/New_York`. If the literal abbreviation in the file disagrees with the computed status (e.g. file says `EDT` but date is 2026-12-15, which is EST), either:

\- **Reject the row** with a clear error citing the mismatch (strict), OR

\- **Warn and proceed** using the computed offset (lenient, but logs the discrepancy)



Decision between strict/lenient deferred until a real user reports a mismatch — first complaint is the signal, not speculative defense.



Blocks: nothing in v0.2.0. Fallback behavior (trust the literal) is correct in 99%+ of real cases. Only matters if Webull ships stale abbreviations across DST boundaries; no evidence yet that they do.



Estimated effort: <1 day (computed offset is a one-liner with `Intl.DateTimeFormat` or a tiny lookup table for US DST transitions).

\### Store true UTC timestamps + rename column + add display-local rendering

Source: Day 4 design decision (2026-05-16). The column `executions.timestamp_utc` (`schema.ts:69`) is misnamed — all four DAS parsers and the new Webull parsers all store bare local Eastern, not UTC. Cross-format sort works because everyone agrees on the convention, but the column name lies and users in non-Eastern timezones (international users) can't currently see times in their own local time.



Audit evidence:

\- `parse-tradehistory.ts:64-74` (parseTradeHistoryDate) parses `MM/DD/YYYY` as-is, no TZ math.

\- `parse-tradehistory.ts:77+` (parseTradeHistoryTime) returns `HH:MM:SS` unchanged.

\- `parse-trades-window.ts:65-74` (parseBareTime) returns the literal HH:MM:SS unchanged.

\- `Execution.time` type comment says `ISO YYYY-MM-DDTHH:MM:SS` — no `Z`, no offset.

\- The new Webull parsers (Day 4-5) explicitly drop the `EDT`/`EST` literal and the `+0000` offset to match this convention.



Solution:

1\. Add a parser helper `localEasternToUtc(date: string, time: string): string` and apply to all six parsers (4 DAS + 2 Webull). Output: ISO 8601 with `Z` suffix.

2\. Rename column `timestamp_utc` — keep the name now that it's accurate. (Or rename to something else if the column ever needs to hold pre-UTC data; column rename + data migration in same release.)

3\. Add a display-layer helper that renders UTC in the user's preferred timezone (defaulting to America/New_York for the US market-hours convention most active users will want).

4\. Update all parser tests with hardcoded local Eastern output to assert the new UTC values.

5\. Audit downstream consumers (`electron/stats/`, `electron/analytics/`, `electron/calendar/`, `electron/reports/`, `src/core/insights/`) for any time-based logic that assumed local Eastern. Anywhere that compares against market-hour windows (9:30, 11:00, etc.) must convert UTC → Eastern before comparing — likely the largest part of the work.



Estimated effort: 1-2 days. Touches every parser, every parser test, and any downstream consumer that reads `.time` or `timestamp_utc` directly. Bundle with [[per-date-dst-verification]] since both touch timestamp handling — landing them together saves a second audit pass.



Blocks: Nothing in v0.2.0 — the bare-local-Eastern convention is internally consistent. v0.3.0 cleanup, alongside the dev-DB-isolation and paper-account work as part of the "v0.2.0 left these things half-done on purpose" theme.
