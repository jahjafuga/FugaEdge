\## Strategic / Tentpole Features



\### Migration Importers from Competing Journals

Source: Brendan Hogan (DTSM founder), 2026-05-16



Concept: Build one-click migration importers from Tradervue, TradesViz, Edgewonk, Tradezella, TraderSync. Removes the #1 reason people don't switch journals (years of historical data lock-in).



Why this is bigger than a feature:

\- Switching cost lock-in is incumbent journals' primary moat

\- One-click migration is how Substack beat Medium, Notion beat Evernote

\- Tradervue, TradesViz, Edgewonk all expose CSV/Excel exports

\- Could be the strategic theme of v0.3.0: "Switch from competing journals"



Scope estimate:

\- 1 parser per target platform (4-5 platforms)

\- Mapping logic from each platform's data model to FugaEdge universal model

\- Detection of which platform export user is bringing in

\- Multi-year historical import with progress UI

\- Edge cases for each platform's quirks



Possible home: v0.3.0 (strategic theme)

\### Trade Behavior Analytics — MFE / Reversal Tracking

Source: Mark Douglas, "Trading in the Zone", surfaced 2026-05-17



Concept: 90% of losing trades go in the trader's direction first before reversing. FugaEdge should track Maximum Favorable Excursion (MFE) and time-to-peak for every trade.



Why this matters:

\- Reveals "winners turned losers" pattern across many trades

\- Helps trader understand: setup wasn't wrong, exit was wrong

\- Drives behavior change toward scaling out

\- Mark Douglas attribution adds psychological credibility



Feature scope:

\- For each trade: compute MFE via Polygon historical aggregates

\- Show time-to-peak (how long position was at max profit)

\- Identify "winners turned losers" pattern in Edge Insights

\- Per-playbook MFE statistics

\- Optional: "scale-out simulator" showing what-if you'd taken partial profits at X% gains

\- Educational tooltip with Mark Douglas attribution



Data requirements:

\- Polygon aggregates endpoint (1-minute or 1-second resolution)

\- 1 API call per trade for backfill

\- Async enrichment queue (similar to country/float)



Why not v0.2.0:

\- v0.2.0 focus is broker support + import polish

\- Needs Polygon enrichment to be stable first

\- Deserves dedicated release (not Day 8 add-on)

\- Real depth requires proper UI work



Possible home: v0.4.0 or v0.5.0 — "Trade Behavior" theme release

\### Auto-isolate dev DB from packaged DB (testing infrastructure)

Source: surfaced 2026-05-17 when Day 2 smoke-test data (Dave's CSV scenarios 1-3+6, Brendan's CSVs scenarios 4-5) leaked from `npm run dev` into the real journal dashboard. Required a manual `.bak` restore to recover. Inevitable as testing scales.



Concept: In dev mode (`!app.isPackaged`), use `%APPDATA%\fugaedge-dev\fugaedge.db` instead of `%APPDATA%\fugaedge\fugaedge.db`. Zero-friction separation between dev/test work and real journal data — `npm run dev` automatically writes to a separate DB without anyone remembering anything.



Belt-and-suspenders: add a `FUGAEDGE_DB_PATH` env var override so multiple fixture sets can each have their own DB when needed (e.g., `FUGAEDGE_DB_PATH=...\brendan-fixtures.db npm run dev`).



Trade-off to think through before implementing: this immediately splits the current real-data dashboard from new dev runs. Anyone who currently uses `npm run dev` against real data (e.g., journaling between dev sessions) loses that view unless they launch the packaged installer. If that's acceptable, ship the auto-isolation. If not, ship env-var-override alone first and add auto-isolation later.



Possible home: v0.3.0 (small infrastructure win, frees future smoke testing from data-pollution risk).
