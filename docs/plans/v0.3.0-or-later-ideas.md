## v0.2.1 Candidates — "Analytics polish + UX consistency + Chart overhaul"

Captured from DTSM beta cohort feedback (Tester A, 2026-05-21) during
v0.2.0 development. Held back from v0.2.0 to protect the broker-
import scope, slated for fast follow-up release. Verification
work on 2026-05-22 confirmed each ticket reflects actual codebase
state on feature/v0.2.0-universal-import at commit 1515695.

### Analytics — finer-grained price and share-size buckets
Current buckets ($0-5, $5-10, $10-20 on price; <100, 100-500 on
shares) are too coarse for small-cap momentum traders. Tester A's 99
trades distribute as 66 in $0-5 + 97 in <100 shares — the
analytics tells him nothing he doesn't already know.

Proposed buckets (validated for the Ross Cameron demographic):
- Price: <$2, $2-5, $5-10, $10-15, $15-20, >$20
- Shares: 0-50, 50-100, 100-250, 250-500, 500-1000, 1000-2500, 2500+

Add min-sample-size dim or asterisk on buckets with <5 trades to
avoid misleading "100% win rate" rows on tiny samples.

### Mistakes column + Distance-from-9EMA column on Trades table
Two additive columns, single ticket. Originally triaged as
separate items but they share infrastructure and ship together.

**Mistakes column:** currently mistakes are visible only inside
the Trade Detail Modal. When users filter the table to MISTAKES,
they expect to see which mistakes are tagged at-a-glance. Render
mistake chips with compact overflow ("first 2 chips + +N more").

**9EMA distance column:** originally parked at v0.3.0 thinking new
infrastructure was needed. Verification 2026-05-22 confirmed the
data already exists — the existing `trade.entry_ema9_distance_pct`
field (backfilled by `computeEma9Distance`, computed on 1-minute
bars) powers the Trade Detail "ENTRY VS EMA9" stat card. Just
surface it as a sortable column. Match the label convention used
by the v0.2.0 EMA9 rename ticket — likely "9EMA Δ (1m)" or similar.

### Cycle through trades inside Trade Detail Modal
Current behavior: must close modal and click next trade to
navigate. Add ← → keyboard navigation + on-screen prev/next
buttons in the modal chrome. Cycling respects the current
filtered list (e.g. if filtered to MISTAKES, cycle goes through
the 3 mistake-tagged trades, not all 99). Boundary handling:
first trade → prev disabled; last trade → next disabled.

### Card click → Trade Detail Modal in Charts and Grid views
UX inconsistency: Table rows are clickable to open the modal,
Charts/Grid cards are not. Add the same onClick → modal handler
used by the Table row to both Charts cards and Grid cards. Uses
existing modal infrastructure and existing data — purely
affordance work.

### Scratch trade range configurable

**Largely OBE by v0.2.3 (2026-06-02).** The hardcoded ±$2 band was REMOVED — scratch
is now `|net_pnl| ≤ 0.005` (strict-zero + tolerance), so the core complaint below
(±$2 swallowing small-share traders' real W/L) is already fixed: a $1.50 trade now
counts as a real winner on every surface. A user-configurable scratch *tolerance*
remains a possible future feature, but it is no longer fixing a misclassification
bug — the default would be the 0.005 epsilon, not $2, and most traders want $0.00
strict. Re-scope before building. Original ticket kept below for history.

Current (pre-v0.2.3): a trade with net P&L between −$2 and +$2 is classified
as a "scratch" (hardcoded). For small-share-size traders (Tester A
trades 28 shares; many DTSM members start at 50-100), ±$2
swallows a meaningful share of real winners/losers — distorting
win rate, profit factor, and Edge Insights computations
downstream. 30 of Tester A's 99 trades are misclassified.

Fix: Settings → Risk Parameters → "Scratch trade range: ±$___".
Default $2 (preserves current behavior). New SQLite column on
user_settings, schema migration with proper test coverage. Trade
classification should remain a pure function in /src/core
applied at query time, not stored as denormalized boolean —
verify this is the case before claiming "threshold changes
retroactively reclassify all history."

Open question for Tester A: absolute-dollar threshold (±$X),
percentage of account size (0.05% of account), or both? Ship
absolute-dollar first.

### Winners Turned Losers panel on Deep Analytics > Execution
Tester A asked for a "Gave Back Profits" panel similar to "Money
Left on Table." On the math, those would mostly overlap. The
valuable separation is **trades that were winners but became
losers**: peak P&L during trade was positive (above threshold
like $0.05/sh or 1% of entry), final P&L < 0. The Mark Douglas
"winners turned losers" pattern that's psychologically
destructive.

Ship under the clearer name "Winners Turned Losers" rather than
Tester A's "Gave Back Profits" framing. Same data source as existing
Money Left on Table panel — MFE infrastructure already in place
(38/99 of Tester A's trades currently have computed MFE; expanding
to 99/99 is separate work, see v0.3.0 entry below).

### Chart UX overhaul — DAS-style execution markers
Source: Tester A (DTSM beta cohort), 2026-05-21, with concrete DAS
Trader Pro reference screenshot.

Current FugaEdge chart renders execution markers as large arrows
with full-text labels ("S 2 @ 7.75", "S 6 @ 7.0?") stacked
vertically above/below the price action. On multi-fill trades
(every Ross Cameron strategy trade, basically) the labels
overlap and truncate — Tester A's RUBI screenshot shows 6 sell-fill
labels colliding inside a 3-minute window. The chart becomes
unreadable.

DAS Trader Pro convention (Tester A's reference screenshot, CODX
1-min chart):
- Small triangle markers, not labeled arrows. Up-triangle for
  buys, down-triangle for sells. Color-coded.
- Triangles placed at execution time/price, snug to candles
  (not stacked in a separate vertical band).
- No persistent text labels. Multiple fills inside the same
  minute = multiple triangles stacked at that candle's
  position. Hover or click reveals fill detail
  ("S 100 @ 6.74").
- No horizontal entry/exit reference lines — triangles do the
  job.

This convention is established across DAS, Tradervue, TradesViz,
and most other journals because it scales to multi-fill trades.
FugaEdge's current rendering scales poorly past 2-3 fills.

Bundles with the parked "Real intraday 1-min charts on Charts
view cards" entry below — both touch the same chart rendering
surface. If you build the Trade Detail chart overhaul properly,
the Charts view cards can pick up the same rendering for free.

### Tradervue parity stats — Reports and Deep Analytics
Source: Tester A (DTSM beta cohort), 2026-05-21, with Tradervue
screenshots as reference.

Six new stats, all computable from the existing trades table
with no new enrichment. Add to the appropriate Deep Analytics
or Reports tabs (don't dump as one new panel):

- **Average Daily Volume** — avg shares traded per day
- **Average Per-share Gain/Loss** — net P&L / total shares,
  summed across all trades
- **Average Hold Time bucketed by outcome** — separate metrics
  for scratch / winning / losing trades. Tradervue surfaces
  these as a row of 3.
- **Max Consecutive Wins / Losses** — streak data. Define
  scratch-handling rule explicitly (does a scratch break a
  streak or preserve it? Tradervue's convention is unclear).
  Document the chosen rule in the tooltip.
- **Avg Win % / Avg Loss %** — percent-based versions of the
  existing "Average Winning Trade" / "Average Losing Trade"
  dollar stats. Use the percent() helper added in commit
  853b2ce.
- **Max Win % / Max Loss %** — percent-based versions of
  Largest Gain / Largest Loss.

Surface placement: Per-share P&L and Win/Loss % stats belong on
Deep Analytics > Performance. Hold Time bucketed belongs on
Psychology or Execution. Max Consecutive belongs on Streaks
(separate from existing Discipline Streak — note this is a
*P&L outcome* streak, not a *discipline rule-following* streak).
Don't ship them as one undifferentiated panel; consider the
information architecture.

\## Strategic / Tentpole Features

### Point-in-Time Float — Float as a First-Class Momentum Pillar
Source: surfaced 2026-05-18 during Day 7B float-display bug diagnosis

Possible v0.3.0 tentpole theme: "FugaEdge knows what the float actually was when you traded it."

Concept: For every trade, store and display the actual tradable share count (float) **as it was on the date of the trade** — not today's number, not shares outstanding, not a current snapshot. Treat float as a first-class momentum-trading pillar alongside catalyst, time of day, and price action.

Why this is bigger than a feature:
- Float is one of the core pillars of momentum stock selection (Ross Cameron strategy, the entire DTSM playbook)
- Small-cap momentum names reverse-split, dilute, and re-issue shares constantly — today's float says nothing about what the setup looked like on the day of the trade
- A correct point-in-time float unlocks the Float Bucket Edge Insights idea already in this file, the MFE/reversal tracking idea, and any future dilution-detection or offering-flag features
- No competing journal does this well — most show today's float, or shares outstanding mislabeled as float

What we discovered on 2026-05-18:

FugaEdge v0.1.6 / early-v0.2.0 was displaying shares outstanding (from Polygon's `share_class_shares_outstanding` field via `fetchTickerDetails`) and labeling it "Float." The codebase already had a `fetchFreeFloat` function targeting Polygon's `/stocks/vX/float` endpoint, written and documented but never wired up (zero callsites outside its own definition file).

3-ticker curl test against Polygon's `/stocks/vX/float` on the free tier:

| Ticker | Real float (Finviz) | FugaEdge displayed | Polygon `/stocks/vX/float` returned | Notes |
|---|---|---|---|---|
| CISS | 520K | 528K | 2,432,345 | ~4.7× too large; effective_date 2026-03-19 |
| MOBX | 8.72M | 10M | 80,797,136 | ~9.3× too large; effective_date 2026-02-17 |
| LABT | 6.09M | 14M | `results: []` | empty response |

Conclusions:
- The dedicated free-float endpoint exists on the free tier but returns data that is *worse* than what FugaEdge currently shows
- The ratios (~5× CISS, ~10× MOBX) are suspiciously close to clean integers — likely stale, split-unadjusted data
- The `vX` in the URL is a signal: not `v1`/`v2`/`v3`, likely an experimental or low-priority endpoint
- LABT has no float data at all — coverage on small-caps is patchy
- Wiring `fetchFreeFloat` as-is would make the displayed number significantly worse than the shares-outstanding figure that's currently shown

Conclusion: a free-tier Polygon endpoint won't solve this. Point-in-time float needs a real data-sourcing plan.

Data-sourcing options to evaluate (v0.3.0 planning):
1. Polygon paid tier — verify the float endpoint is materially better on paid before committing
2. SEC EDGAR scraping — parse Form 10-Q/10-K/8-K share-count disclosures by filing date; free but multi-day build
3. Alternate vendor (Benzinga, Intrinio, Finnhub historical) — cost vs. coverage trade-off
4. Hybrid — use Polygon's `free_float_percent` × ticker-details shares-outstanding as a derived float; cheap, validated against 2/3 tickers today but needs a larger sample (10+ tickers) before trusting

Good news on the technical side: the DB infrastructure already exists.
- `trades.float_shares` column was added in v0.2.0 Day 1 as a per-trade snapshot (currently populated with shares-outstanding-at-import-time, but the column shape is right)
- The orchestrator pattern (`src/core/float/orchestrator.ts`) is in place
- The user-override path (`FloatEditor.tsx`, `tradeFloatSave` IPC) is in place
- v0.3.0 doesn't need a schema migration; it needs the correct historical value to flow into the existing column

v0.2.0 mitigation (shipped on Day 7C, 2026-05-18):
- Disabled the `runFloatSweetSpot` Edge Insight rule (made analytic claims based on shares-outstanding bucketing)
- Hidden the `FloatBreakdownCard` (subtitle promised "tradable share float at the time of the trade" which was false)
- Relabeled display surfaces from "Float" to "Shares Out" across TradesTable, TradeDetailModal, FloatEditor, DayTradesPanel, Trades.tsx column toggle
- Dropped the word "float" from soft-claim copy in Hot Symbol rule body, Playbook placeholder, onboarding tour, VolumeTab caption
- Release notes name the v0.3.0 point-in-time-float feature as the proper fix

v0.3.0 scope when this ships:
1. Pick a data source from the four options above (decision-tree, not all-of-the-above)
2. Build the enrichment path that fetches historical float for the trade date
3. Re-enable `runFloatSweetSpot` with correct point-in-time data
4. Re-enable `FloatBreakdownCard` with subtitle that's actually true
5. Restore "float" wording across the four soft-claim surfaces that v0.2.0 softened
6. Resolve the two collateral findings v0.2.0 deferred:
   - Bucket-boundary inconsistency between Analytics (4 buckets) and Reports (3 buckets) — unify on a single canonical scheme
   - Per-trade-vs-per-symbol read inconsistency — Analytics reads from `trades.float_shares`, Reports reads from `market_data.float` cache; user overrides only show on one side. Unify to per-trade read everywhere once the trade-level value is correct.
7. Rename DB columns and TypeScript field names from `float` / `float_shares` to something that disambiguates point-in-time-float from current-shares-outstanding (e.g., `float_at_trade_date` and `shares_outstanding_current`)
8. Build the Float Bucket Edge Insights ideas this unlocks (separate parking-lot entry)

Estimated effort: 2-3 weeks depending on data-source choice. SEC EDGAR path is the longest; Polygon paid tier path is the shortest (if it works).

Dependencies satisfied by this feature: Float Bucket Edge Insights, dilution detection, MFE/reversal tracking (all elsewhere in this file).

Possible release home: v0.3.0 tentpole, or shared v0.3.0 theme with shadcn migration / timezone architecture / i18n if those land together.

- Empirical evidence added 2026-05-19 — CODX comparison: Polygon's free-tier "shares outstanding" returned 3.60M; the real float (verified against a separate professional data source) was 3.42M. That's a ~5% gap on a single mid-sized ticker, confirming the v0.1.6→v0.2.0 finding that what Polygon's free tier returns as "shares outstanding" is structurally not the same as true tradable float. Strengthens the case for point-in-time float being a v0.3.0 tentpole feature, not a quick fix.

\### 5 Pillars of Stock Selection — configurable trade-quality framework

Source: Lao, 2026-05-17 (refined during Day 7A scope discussion)



Concept: A configurable framework where users define their 5 (or N) pillars of stock selection — the criteria they use to qualify a momentum setup before entry. Default preset inspired by Ross Cameron / Warrior Trading public criteria. Users can edit, reorder, or replace pillars to match their own edge.



The 5 default Pillars (as Lao defined them):

\- Relative volume (RVOL ≥ 5x the 30-day average)

\- Float (low float preferred for momentum)

\- Price range (typical setup price window)

\- Catalyst (news, earnings, halt resume, etc.)

\- Daily change percentage (gap, % from open, % from premarket)



Why this matters:

\- Ross's Warrior Trading audience already thinks in terms of A+ setups vs lower-quality setups. A pillar framework formalizes that intuition.

\- FugaEdge is positioned for momentum traders — having a built-in selection framework that mirrors how this audience already trades is identity-aligned.

\- Configurability means it works for traders who use other strategies (Tester B's pillars may differ from Tester A's may differ from Lao's).

\- Could become the spine of a richer Edge Insights surface: "Your win rate by pillar-count," "Which pillar matters most to your edge."

\- This idea is what drove the Day 7A Commit 5.5 decision to restore aggregates fetching at import time — without daily_volumes + avg_volume, the RVOL pillar can't score, which would break the feature on day one.



Data dependencies (now ready as of v0.2.0):

\- RVOL: needs `daily_volumes` + `avg_volume` (populated at import as of v0.2.0 Day 7A Commit 5.5)

\- Float: needs `float` (populated at import as of v0.2.0 Day 7A Commit 5 + bug fix)

\- Price range: derivable from trade entry/exit prices (already in trades data)

\- Catalyst: currently manual user tagging; future Polygon News API integration (separate v0.4.1+ ticket)

\- Daily change %: needs intraday open / previous close (requires Polygon aggregates extension)



Direction unresolved — needs decision before scoping:

\- Journal direction: retroactive scoring of imported trades. For each trade, surface which pillars were/weren't present. Edge Insights then correlates pillar-presence with outcome.

\- Screener direction: real-time pre-trade scoring. User enters a candidate ticker during the session, app computes pillar score from live data, helps with go/no-go decision.

\- Hybrid: define pillars once, use them both pre-trade (manual entry) and post-trade (auto-scored on import via Polygon data).



Scope estimate (large, regardless of direction):

\- Pillar definition schema + storage (per-user, editable)

\- Default preset based on Ross's criteria (research required for exact thresholds)

\- Scoring logic per pillar (some auto-computable from Polygon — float, RVOL, daily change; some require user judgment — catalyst quality, chart cleanliness)

\- UI for editing pillars in Settings

\- UI for displaying pillar scores on trade cards

\- Edge Insights rules that use pillar data

\- Onboarding to set defaults



Open questions:

\- v0.3.0 tentpole or v0.4.0+ feature? Likely v0.4.0 — it's a real product surface, not a polish item, and benefits from shadcn migration landing first.

\- Pairs with the existing playbook tag system, or replaces it? Playbooks describe setup type; pillars describe setup quality. Possibly orthogonal, possibly overlapping.

\- Pro tier feature, or free? Probably free for default preset, Pro for custom pillars + advanced insights — TBD.



Risks:

\- Ross attribution: using his framework directly may need permission or careful framing. "Inspired by Ross Cameron's criteria" vs "Ross Cameron's pillars" is a meaningful legal/relationship distinction.

\- Scope: this is a feature, a UI surface, an analytics surface, and an onboarding flow combined. Not a single-day build.



Possible home: v0.4.0+ (deserves its own theme release, not a side-quest)

### Intraday technical-indicator pipeline
Source: Tester A (DTSM beta cohort), 2026-05-21. Multiple related
requests bundled here.

Concept: Pluggable framework for per-trade enrichment with
technical indicators computed from intraday OHLC data. Each
indicator implements a standard interface: value-at-entry,
distance-from-value, boolean signal state (e.g. "MACD had
crossed bullish within last N bars"). UI surfaces: per-trade
card stats, optional Trades table columns, Edge Insights signal
rules. Settings let user pick which indicators to track so we're
not computing everything for everyone.

Initial indicators that map to user requests:
- 9EMA (1-min) — already partially shipped via
  `computeEma9Distance`. Would become the reference
  implementation.
- MACD (Tester A): capture whether trade was taken at a 1-min MACD
  bullish crossover.
- Daily SMA distance (Tradervue Advanced Filters convention
  surfaced via Tester A's screenshot): same indicator family as
  9EMA but on the daily timeframe — useful for "extended from
  daily SMA" context.
- VWAP (likely next ask — Ross Cameron's playbook uses it
  heavily).
- RSI, Bollinger Bands as common follow-ups.

*(Note: MACD-at-entry, VWAP-at-entry, and 9/20 EMA-at-entry parts of this pipeline were realized in v0.2.4's Technical Analysis tab. The pluggable framework, RSI, Bollinger Bands, Daily SMA distance, and Real 1-min charts on Charts cards remain v0.3.0+ scope.)*

Why bundle: Tester A has requested four indicator-aware features
across two batches of feedback (9EMA column, 9EMA label,
MACD crossover, daily SMA filter). Treating each indicator as
its own feature pulls into one-at-a-time work indefinitely. One
framework, many indicators, cleaner long-term shape.

Also bundles:
- "Real intraday 1-min charts on Trades > Charts view cards"
  (Tester A) — same data source.
- "Chart UX overhaul" (v0.2.1 ticket above) — if v0.2.1 ships
  the Trade Detail chart redesign on top of this framework,
  Charts view cards inherit the same rendering for free.

Open question to resolve before scoping: what's the data source?
Massive (already wired for Trade Detail Chart tab via Polygon
aggregates equivalent, but paid-plan friction), Polygon aggregates
direct (no plan friction but new pipeline), or hybrid?

Scope: Large. Release theme, not a feature. Probably v0.4.0 or
v0.5.0 — "Trade Behavior" or "Technical Indicators" theme
release.

Possible home: v0.4.0+ — bundles with revised MFE / Reversal
Tracking entry (see housekeeping note below).

### Advanced statistical filters (Tradervue Pro parity)
Source: Tester A (DTSM beta cohort), 2026-05-21, with Tradervue
Advanced Filters screenshots as reference.

Concept: A power-user filter surface allowing trades to be
filtered by 20+ statistical predicates with min/max ranges.
Tradervue's Pro-tier filter dimensions (locked behind their
paywall) include:
- Days/Time: day of week, time of day, month, duration
- Price/Volume: price, volume traded, in-trade price range,
  execution count
- Instrument
- Win/Loss: trade result, day result, P&L % gain, P&L (R),

**Strategic note:** Tradervue charges for this feature. That's
market validation that it's a Pro-tier feature. If FugaEdge
eventually has free vs Pro tiers, this is where Pro earns its
money — don't ship as free. This is the natural foundation for
the eventual Pro tier launch noted elsewhere in this doc.

### Technical Analysis Tab — 200 EMA / 200 MA daily-chart distance (deferred from v0.2.4)

Source: Lao, 2026-05-29; partially absorbed into v0.2.4 spec

### Refresh progress bar survives tab switch — NEAR-TERM (Commit P.5, not someday)
Source: Lao observation, 2026-05-29, immediately after Commit P (a98192d) shipped the live refresh progress bar.

Concept: the market/intraday refresh progress bar (Settings) must persist across a tab switch. Today the bar is renderer component-state: navigating away from Settings unmounts the component, the `marketOnIntradayProgress` subscription unsubscribes, and on return the re-mounted Settings has fresh state (`intradayRefreshing=false`, `progress=null`) — so the bar is gone and the button is re-enabled even though the run is still going in main.

Diagnosis (case A, mild — confirmed 2026-05-29): the refresh runs in the MAIN process and completes regardless of tab switch; per-pair `upsertIntradayRow` means no half-written state; the `inFlight` singleton (intraday.ts:44) prevents concurrent double-runs. So there is NO data risk — only the renderer's view of progress is lost. Commit P is honest about every run it shows; this is a surfacing gap, not a correctness bug.

Why near-term, not someday: Tester A's refresh grinds for minutes under free-tier throttling, and users tab away during long runs. On return the bar's gone so it looks stopped — and a re-press starts a fresh (incremental) run rather than resuming the visible one. That partially undoes Commit P's win for exactly Tester A's usage (long runs + tab-switching). Fast-follow to P, not a backlog polish item.

Two fix directions (identified during the P diagnosis):
1. **Hoist refresh state above the router** — keep the refreshing flag + latest progress in app-level state (context/store) that survives route changes, so returning to Settings re-renders the live bar from retained state.
2. **Main exposes queryable running-state + current progress** — a "current refresh status" IPC (or a retained last-progress snapshot) that a re-mounting Settings reads on mount and reconciles with the ongoing `wc.send` stream. Main is the source of truth; the remount rehydrates. This direction also lets the UI correctly show "still running" (button stays disabled) instead of re-enabling mid-run.

Possible home: v0.2.2 fast-follow (Commit P.5) or earliest v0.3.0 — near-term either way.

### Float-override parser UX — silent wipe + missing "use K/M/B" hint — NEAR-TERM, not someday
Source: smoke-found 2026-05-30 alongside the parseInput bare-decimal guard (Commit B).

Two paired UX gaps, both renderer-only, both in `FloatEditor.tsx`:

**Gap 1 — parse failure silently wipes an existing good value.** When `parseInput` returns `null` (bare decimal, garbage like "abc", "1.5%", etc.) and the field previously had a non-null value, `commit()`'s `parsed === value` short-circuit doesn't fire (because `null !== 1500000`), so `onChange(null)` fires → IPC → DB writes null → field clears. **The user's prior good value is destroyed with no confirmation, no error.** This is latent (pre-dates B) but B's bare-decimal guard puts more inputs into the null-return bucket, increasing the surface. The smoke-found case was fresh-trade (prior value null → benign short-circuit branch), but any user editing an already-populated float with a fat-fingered typo hits the destructive path.

**Gap 2 — no "use K/M/B" hint on rejected input.** When `parseInput` returns `null`, the field just goes blank (or stays empty). The user gets no feedback that their typing was rejected, no nudge toward the suffix shorthand that would have worked. Paired with Gap 1, the user can't even tell whether their input was saved or wiped.

Fix (one small commit, ~10-15 lines, renderer-only, no schema):
- Distinguish "user cleared the field deliberately" from "parse failed" inside `commit()`. Track an explicit `intentionalClear` heuristic (e.g. draft was empty before blur) — only on intentional clear does `onChange(null)` fire; on parse failure, revert draft to `int(value)` and DO NOT call onChange.
- Render a small `text-fg-tertiary` hint below the input when the last blur produced a null result on non-empty input: "Use K, M, or B for shorthand (e.g. 1.2M)". Hint clears on next focus.

Why deferred: not destructive on B's smoke surface (fresh-trade case is the benign branch), but it's a real footgun for any subsequent user editing an existing float. Worth a v0.3.0 polish pass alongside the other "v0.2.0 left these things half-done on purpose" UX entries already in this doc.

Possible home: v0.3.0.

\### Auto-isolate dev DB from packaged DB (testing infrastructure)

**PROMOTED to v0.2.1 on 2026-05-22 after failure mode #4 caused user-visible data corruption — see docs/plans/v0.2.1.md.**

Source: three distinct failure modes hit in one evening (2026-05-16) around the dev/packaged-build split sharing `%APPDATA%\fugaedge\`. Each one cost real diagnosis time; the auto-isolation work closes all three.



Motivating incidents:

1\. **Test data leaks dev → real dashboard.** Day 2 smoke-test imports (Tester A's CSV scenarios 1-3+6, Tester B's CSVs scenarios 4-5) ran via `npm run dev` against the shared userData path, polluting the real journal dashboard. Required a manual `.bak` restore from `backups/` to recover.

2\. **Packaged shadows dev (binary-vs-DB-version mismatch).** After the restore, the next launch was the installed packaged build (`%LOCALAPPDATA%\Programs\FugaEdge\FugaEdge.exe`, built 2026-05-14 with `SCHEMA_VERSION='17'`) instead of the dev branch (`SCHEMA_VERSION='18'`). Old binary opened the restored v17 DB, saw "current = target", and silently bypassed the v17→v18 migration. No warning that two binaries with different schema targets were writing to the same file. ~30 min misdiagnosed as a schema-code bug before the file timestamp on the .exe gave it away.

3\. **Env-var bleed across dev tooling.** Running schema diagnostics under Electron's ABI required `ELECTRON_RUN_AS_NODE=1` (the better-sqlite3 binary is compiled against Electron 33's ABI via the `rebuild:sqlite` postinstall, not against system Node). The env var leaked from the manual one-liner into a follow-up `npm run dev`, causing Electron to launch in Node mode and `require('electron')` to return the path string instead of the API namespace — opaque TypeError on `protocol.registerSchemesAsPrivileged`. Addressed by `npm run inspect:schema` (commit fef5b20, cross-env-scoped), but the env-var-bleed-via-shared-shell class is the same dev/test isolation theme.



Concept: In dev mode (`!app.isPackaged`), use `%APPDATA%\fugaedge-dev\fugaedge.db` instead of `%APPDATA%\fugaedge\fugaedge.db`. Zero-friction separation between dev/test work and real journal data — `npm run dev` automatically writes to a separate DB without anyone remembering anything. Directly addresses #1 and #2 above.



Belt-and-suspenders: add a `FUGAEDGE_DB_PATH` env var override so multiple fixture sets can each have their own DB when needed (e.g., `FUGAEDGE_DB_PATH=...\tester-b-fixtures.db npm run dev`). Also closes future "I need to test against Tester A's data without touching dev state" scenarios.



Related: any dev script that needs the Electron ABI should be wrapped in a `cross-env`-scoped npm script (precedent: `inspect:schema` at fef5b20) so `ELECTRON_RUN_AS_NODE` never leaks into the user's shell session.



Trade-off to think through before implementing: auto-isolation immediately splits the current real-data dashboard from new dev runs. Anyone who currently uses `npm run dev` against real data (e.g., journaling between dev sessions) loses that view unless they launch the packaged installer. If that's acceptable, ship the auto-isolation. If not, ship env-var-override alone first and add auto-isolation later.



Possible home: v0.3.0 (small infrastructure win, frees future smoke testing from data-pollution risk; three concrete failure modes already paid the diagnosis tax).

### Windows UIPI dev-shell elevation foot-gun

Source: Day 9 investigation, 2026-05-22.

diagnostic instrumentation — before the elevation mismatch was spotted.

  blocked by Windows UIPI."
- (b) Document the gotcha in the README dev-setup section.
- (c) Add a non-admin PowerShell launch shortcut / script to the repo so the
  default `npm run dev` entry point can't be accidentally elevated.

Relationship to other items: belongs in the same "dev/test environment
isolation" theme as [[auto-isolate-dev-db]] — both are dev-mode foot-guns
that silently waste real diagnosis time and look like product bugs.

Possible home: v0.3.0 — small, paired with the auto-isolate-dev-db work.

\### Per-date DST verification on Webull Mobile timestamps

\### Open-position re-import: purgeOpen wipes rows without re-inserting them

Source: surfaced during Day 8.5 Commit B Gate 5 verification, 2026-05-21.

Re-importing a byte-identical CSV that contains open positions silently purges those rows via `repo.ts` `purgeOpen` without re-inserting them. `commit` runs `purgeOpen` (`DELETE FROM trades WHERE symbol=? AND date=? AND is_open=1`) for each affected `(symbol, date)` before the insert loop — but `annotateTripStatus` has already flagged the re-parsed open trips as `duplicate` (their `exec_hash` is unchanged), so the insert loop skips them. Pre-existing behavior: `repo.ts` is not touched by Commit B (`git diff --name-only` on the Commit B changeset confirms it) — the timezone flip merely happened to be the work that exercised this path. Decide for v0.3.0:
- keep current behavior (re-importing an identical open-position file loses those rows), or
- switch `purgeOpen` to also re-insert the open trips from the parsed batch, or
- skip `purgeOpen` entirely for duplicate-only batches (when every trip is annotated `duplicate` there is nothing to wipe-and-rewrite).

Possible home: v0.3.0, alongside the Trade Deletion / Import Session Management work — both touch import/commit semantics.

### Backup-latch coordination — make the two v0.2.0 backup latches aware of each other
Source: bullet 5/6 Q6 investigation, 2026-05-25, during DB migration verification planning.

Concept: v0.2.0's migration code creates two safety backups via two independent latches: `maybeBackupForV020` (fires when `schema_version < 18`, creates `pre-v0.2.0-<ts>.bak`) and the tz-utc migration's own pre-migration backup (creates `pre-v0.2.0-tz-<ts>.bak`). Neither latch knows about the other.

Edge case the lack of coordination produces: if `maybeBackupForV020`'s latch was already set on a previous launch (e.g. a partial migration that crashed after the backup but before the schema bump completed), the next launch sees the latch set and skips creating a fresh `pre-v0.2.0-<ts>.bak` even though the schema is still pre-18. The tz-utc backup will still fire normally, so the user has one safety net — but it's `pre-v0.2.0-tz-*.bak` reflecting a partial schema state, NOT a clean pre-v0.2.0 starting point.

Why it matters: probably fine in practice (SQLite transactions make the partial-crash mode unlikely; we have no evidence of it happening). The safety net exists either way. But if a user ever needs to roll back from a v0.2.0 install to the original v0.1.6 state, they need to know which .bak to restore from. The v0.2.0 release notes now disclose which .bak corresponds to which state, which is cheap-mitigation enough for ship.

Two solution shapes: (a) make the latches aware of each other — add a "pre-v0.2.0 backup exists" marker independent of the latch; if the latch says "fired" but no `pre-v0.2.0-*.bak` is on disk, fire the backup again; (b) document precisely which .bak corresponds to which state in user-facing materials. Cheaper.

Estimated effort: (a) 1-2 hours including a test for the partial-crash replay path; (b) zero, already done in v0.2.0 release notes.

Possible home: v0.3.0 if (a). Otherwise the documentation note already exists.

Blocks in v0.2.0: nothing. Probably-fine edge case + release-notes disclosure is sufficient mitigation for ship.

### Open-position P&L uses gross differential, not true mark-to-market unrealized
Source: Day 12 smoke test, 2026-05-26, fixture 3 (Tester A's open positions).

Concept: When a round trip is partially closed (e.g. bought 56 shares, sold only 28) or short-but-uncovered (sold short, never bought back), v0.2.0 computes P&L as `sells_value - buys_value` regardless of open status. UI labels this "unrealized P&L" but the formula is a gross differential, not true mark-to-market.

Example from Tester A's data:
- HTCO long: bought 56 @ $9.295 = $520.52 spent, sold 28 @ $11.24 = $314.72 received. Stored pnl = -$205.80. This treats the unsold 28 shares as if their cost basis is fully realized and recovered nothing — equivalent to "what you'd lose if the position were liquidated at $0 right now."
- HCTO short: sold 28 short @ $6.47 = $181.16 received, bought 0 = $0 paid. Stored pnl = +$181.16. Treats the uncovered short as if shares were obtained at zero cost.

The sign is directionally correct (HTCO with unsold-shares-not-recovered = paper loss; HCTO with short-still-open = paper profit if price drops). The magnitude is "liquidate-at-$0" not "liquidate-at-current-market."

Practical implication: open positions appearing in Dashboard aggregations like "Largest Winner" and "Largest Loser" show as the extreme values in the dataset — not because they're the biggest realized P&L, but because the open-position formula gives them outsized magnitudes. Today's smoke test confirmed Tester A's Largest Winner ($181.16 HCTO) and Largest Loser (-$205.80 HTCO) are both open positions, despite his actual realized trips having much smaller magnitudes.

True mark-to-market would require live market quotes (Polygon, broker API, or end-of-day close prices). Available in scope of v0.4.0 or later when live-data integration is a real feature.

Companion concern: aggregations should likely filter `WHERE is_open=0` for "realized" stats and treat opens separately. Today's smoke test confirmed Dashboard's Largest Winner/Loser does NOT filter — opens are included. This is a separate code-audit task for v0.3.0 work.

Scope estimate: medium. Requires live-quote integration for true mark-to-market (large) OR audit + filter of aggregation sites to exclude opens from realized stats (smaller). The is_open filtering audit alone is 1-2 days.

Possible home: v0.4.0 for mark-to-market; v0.3.0 for is_open aggregation audit.

### SQLite WAL-housekeeping bumps mtime on clean Electron shutdown without writes
Source: Day 11 push-verification investigation, 2026-05-25, after a manual visual-verification app launch produced an unexplained mtime bump on the live DB.

Concept: Opening a SQLite database in WAL mode through a packaged Electron app and closing it cleanly bumps the file's mtime even when zero write transactions occur. The mechanism: SQLite checkpoints the WAL on clean shutdown, which touches a small region of file-internal metadata (e.g. checkpoint salt update around byte offset 65516 in the relevant page). The SQLite file-change counter in the database header (bytes 24-27) is NOT incremented — that counter only ticks on actual write transactions. So content integrity is preserved while filesystem mtime appears modified.

Practical implication observed: bullets 5/6 verification's check-5 step compared live %APPDATA% mtime against a "before" baseline and flagged a 12:35:20 bump that was actually a harmless manual visual-verification launch by Lao. ~10 minutes of forensic investigation to confirm Cat 1 (authorized read-only app launch) versus Cat 2 (write event) versus Cat 3 (filesystem-level touch). The investigation was worth doing — but the trigger was a false positive on a naive mtime check.

Why it matters: any automated test or CI check that uses mtime as a "did the DB change?" signal will produce false positives whenever a user opens FugaEdge against that DB. Same goes for backup tooling that does "only back up files modified since last run" — those will repeatedly back up DBs that haven't actually changed content.

Correct integrity signals (in order of strength):
- (1) SQLite file-change counter (DB header bytes 24-27). Increments on every write transaction. Dispositive.
- (2) Byte-level hash comparison. Catches everything but takes longer than (1).
- (3) Row-count + sample-content read-only SQL queries. Catches all real data changes.
- DO NOT use mtime alone as an integrity signal.

Scope estimate: zero engineering effort. Documentation / testing-procedure note for the same testing playbook target as tickets 4, 5, 6 from bullet 7.

Possible home: v0.3.0+ documentation pass alongside tickets 4-6. No code change required.

Related sub-finding (Day 12 smoke test, 2026-05-26): a corollary to this WAL behavior surfaced during fixture 4. When the Electron app's GUI window closes but the process tree doesn't fully terminate (renderer/GPU/utility processes lingering), no WAL checkpoint fires because the last DB connection hasn't closed. The main fugaedge.db file stays at its pre-import size; the fugaedge.db-wal sidecar holds the new data. SQL snapshots taken in this state appear to show pre-import data even though import logs prove import succeeded. Same root cause (WAL checkpoint timing); different presentation. Mitigation in testing procedures: explicitly verify `Get-Process FugaEdge` returns empty before snapshotting; fall through to `Stop-Process -Name FugaEdge` if needed. Future migration/smoke test scripts should bake this in as a procedural step.

THE journal for Warrior Trading / small-cap-runner / momentum-

### Esc double-closes stacked modals (inner + outer together)
Source: v0.2.3 Phase 3, 2026-06-01 (surfaced by the soft-delete ConfirmModal).

Concept: when a ConfirmModal (or any Modal) is stacked over another modal, both
register their own `document` keydown listeners, so pressing Escape fires both —
the inner dialog AND the outer modal close at once. Surfaced by the P3
soft-delete flow (ConfirmModal over TradeDetailModal on the Trades page), but
it's Modal stacking behavior, not introduced here.

Why it matters: Esc-to-cancel on a nested confirm is a reasonable reflex; today
it also dismisses the trade modal behind it. Low severity (Cancel button +
backdrop click are unaffected; only the Trades page has a stacked-confirm path
in P3) but a papercut that spreads as more modals stack.

Scope: small, renderer-only. Have the inner Modal's Esc handler call
`stopImmediatePropagation()` (not just `stopPropagation()`) so sibling
document-level listeners don't also fire; or track a modal stack and let only
the topmost handle Esc.

Possible home: v0.3.0 polish (pairs with the Trash card work that adds more
stacked-confirm surfaces).

### Soft-delete failure path swallows the error silently
Source: v0.2.3 Phase 3, 2026-06-01.

Concept: TradeLifecycleFooter's `handleSoftDelete` awaits `onSoftDelete` in a
try/finally with no catch, and ConfirmModal (lightweight by design) has no error
slot. If `ipc.tradeSoftDelete` rejects, the `finally` clears `busy`, the
ConfirmModal stays open with no message, and the user gets no signal the delete
failed. (Reset journal's heavy TypeToConfirmModal DOES surface errors; the
lightweight path intentionally does not.)

Why it matters: a failed move-to-Trash currently looks like nothing happened.
Rare in practice (local SQLite, version-bump IPC) but a silent failure on a
destructive-leaning action is the wrong default.

Scope: small. Either surface a toast on rejection from the page handler, or give
ConfirmModal an optional inline error slot the footer populates on catch.

Possible home: v0.3.0 polish (alongside the Trash card / Delete Forever work,
which will want the same error-surfacing).

### Delete-Forever attachment-cleanup partial failures aren't surfaced
Source: v0.2.3 Phase 5 recon + build, 2026-06-02.

Concept: the hard-delete IPCs (`tradeHardDelete` / `tradesHardDeleteBulk`) return
`void`. The DB delete is atomic; the on-disk attachment cleanup is fail-soft and
logged in the MAIN process only (`removeAttachmentsOnDisk` swallows each `rm`
error, never throws, never rolls back). So if N trades hard-delete but some
attachment files can't be removed from disk, the renderer cannot tell the user
"N deleted; M files could not be removed" — that information never crosses the
IPC boundary. P5 deliberately did NOT invent a return-shape change for this (out
of P5 scope; an honest gap beats a fake toast we can't back with real data).

Why it matters: orphaned attachment files leak silently on a permanent-delete
action. Rare (local fs, files usually present) but a real cleanliness/correctness
gap.

Scope: small-to-medium. Widen the hard-delete IPC return shape to carry an
attachment-cleanup result (e.g. `{ filesRemoved, filesFailed }`), thread it to
the Trash UI, and surface a non-blocking notice on partial failure.

Possible home: v0.2.4 — pairs with the auto-purge work, where the IPC layer
naturally evolves (auto-purge also hard-deletes + cleans attachments, so it will
want the same cleanup-result surface).

### v0.3.0 Security Hardening Pass — full scope

Source: read-only security audit run 2026-06-02 against the live `feature/v0.2.3-dave-feedback` branch. 14-item Electron-specific posture review (separate from the v0.4.0+ webapp design plan at `docs/plans/security-design-v0.4.0.md`).

Concept: FugaEdge's current desktop security posture is structurally sound — 8/14 audit items PASS, 0 FAIL, 6 WARN. The WARNs are hardening opportunities, not vulnerabilities. This entry consolidates them into a single coordinated v0.3.0 work stream. Each Tier 1 item is its own real risk reduction; Tier 2 is good hygiene; Tier 3 is small wins.

**Tier 1 — biggest real impact:**

- **Code signing for Windows installer.** Currently unsigned (`electron-builder.json` win target has no `certificateFile` / `certificateSubjectName`). Real attack: GitHub account compromise → malicious release → auto-updater verifies SHA512 against attacker-controlled `latest.yml`, ships malware to every beta user. Signing breaks this because a malicious binary has wrong/no publisher cert. Cost: ~$200/year standard cert (DigiCert/Sectigo) + ~1 day to wire into electron-builder. Also resolves Windows SmartScreen "unknown publisher" friction. Decision settled: buy standard cert (not EV — EV requires hardware token, friction for solo dev) when starting v0.3.0 security pass, not before (annual billing starts on issue).
- **API key encryption at rest.** Polygon/Massive (`polygon_api_key`) and FMP (`fmp_api_key`) sit plaintext in SQLite settings table (`electron/settings/repo.ts:70-71, 118-123`). Migrate to OS keychain via Electron `safeStorage.encryptString()` (Credential Manager on Windows). One-shot migration on launch, identical pattern to v0.2.3 schema-25 MFE reset. Decrypt only in-memory when calling provider. Cost: ~1 day.
- **Tighten CSP.** Renderer index.html CSP allows `'unsafe-inline'` and `'unsafe-eval'` in script-src. `'unsafe-inline'` is there for the theme-bootstrap script (prevents flash-of-wrong-theme); fix is to move that to an external file with nonce. `'unsafe-eval'` likely Vite — verify it can be removed in production build. Cost: 1-2 days.

**Tier 2 — good hygiene:**

- **Sandbox enable + preload rework.** `sandbox: false` in BrowserWindow (`electron/main/index.ts:67`) — comment says "preload needs CommonJS require." Flipping to `sandbox: true` requires refactoring preload for ESM/sandboxed pattern. With `contextIsolation: true` and `nodeIntegration: false` already in place, blast radius is already small, but this closes the gap. Cost: 1-2 days.
- **Runtime schema validation on IPC handlers.** All 60+ ipcMain.handle channels rely on TypeScript types (erased at runtime). Settings/repo.ts has defensive coercion; most other handlers don't. SQL is parameterized throughout (verified — no injection risk), so this is hardening for a future compromised-renderer scenario, not a present vulnerability. Add Zod schemas to every handler. High carry-over value: same schemas reusable as Layer 3 input validation in v0.4.0+ webapp. Cost: 2-3 days.
- **Move API keys from query-string to headers.** Currently sent as `?apiKey=...` to api.massive.com and financialmodelingprep.com. FugaEdge code is careful to log only paths, never full URLs, but keys can land in upstream proxy/ISP logs out of our control. Both providers support Authorization headers. Cost: <half day.
- **`.gitignore` gaps.** Add `*.env` glob (current `.env*` would miss `prod.env`) and `/backups/` rule. Cost: 30 seconds. Could ship anytime as a tiny standalone commit; not actually gating anything else.

**Tier 3 — small wins:**

- **Window-open scheme validation.** `electron/main/index.ts:85-88` passes any URL from `window.open` to `shell.openExternal` without scheme check. The IPC `APP_OPEN_EXTERNAL` path correctly validates `http:/https:` (`index.ts:112-122`); make the window-open handler consistent. Cost: 3-line fix.
- **Error monitoring with PII scrubbing.** Sentry or equivalent. Hard requirement: scrub trade data, P&L, API keys from reports (default-included otherwise). Cost: ~1 day.
- **`npm audit` + Snyk on CI.** Current `npm audit --omit=dev`: 0 vulnerabilities — but a CI gate prevents future regressions. Fail build on `critical`, warn on `high`. Cost: half day.

**Total v0.3.0 security pass scope:** ~5-7 focused days. Budget the cert cost ($200) separately.

**Audit findings that came back PASS (no action needed):** `contextIsolation: true`, `nodeIntegration: false`, no `ipcRenderer` direct exposure, no native modules in renderer, no `child_process`/shell exec, no real credentials in git history, 0 npm prod vulns, no telemetry, no webview/iframe/BrowserView, parameterized SQL throughout. Foundation is sound.

Scope: large — 5-7 days, several independent sub-streams. Don't bundle into a feature release; do as its own focused work stream.

Possible home: v0.3.0 — Security Hardening Pass. Schedule before broader beta cohort expansion. Code-signing cert is the long-pole item (annual billing, identity verification ~1-3 business days).
