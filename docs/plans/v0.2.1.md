## Auto-isolate dev DB from packaged DB (testing infrastructure)

**PROMOTED to v0.2.1 on 2026-05-22** from the v0.3.0 parking lot, after
failure mode #4 of this exact hazard caused user-visible data corruption.
The original entry remains in `docs/plans/v0.3.0-or-later-ideas.md` (with a
back-pointer to this file); it is reproduced in full below.

### Why v0.2.1 — not v0.3.0, not v0.2.0

The hazard — a dev build and a packaged build both writing to the single
shared `%APPDATA%\fugaedge\fugaedge.db` — has now produced four failure
modes. The first three (2026-05-16) were diagnostic friction: lost dev
time, nothing the user saw. **Failure mode #4 (2026-05-22) corrupted
user-visible data in the founder's own real journal** — the stale May-14
v0.1.6 packaged `FugaEdge.exe` imported two LFS trades as bare-local-Eastern
timestamps into a database current Commit B code reads as true UTC, leaving
those rows permanently displayed ~5 hours off (full detail: the "LFS trades
12 & 13 — bare-local timestamps left unfixed" entry in
`docs/plans/v0.3.0-or-later-ideas.md`).

v0.2.0 is going to beta testers — Tester A, Tester B, the Webull user — who will
each plausibly run both a dev build and a packaged build against the same
userData path. The hazard is no longer internal dev friction; it is a
data-integrity risk for every beta tester. That is what lifts it out of the
v0.3.0 parking lot.

It does **not** belong inside v0.2.0 itself: that release just shipped
Day 8.5, the schema-19 migration is settled, and a finished release should
not be reopened for scope expansion. A quick follow-up patch — v0.2.1 — is
the correct home.

### The entry (copied from the v0.3.0 parking lot)

Source: three distinct failure modes hit in one evening (2026-05-16) around
the dev/packaged-build split sharing `%APPDATA%\fugaedge\`. Each one cost
real diagnosis time; the auto-isolation work closes all three. A fourth
failure mode (2026-05-22) later escalated the hazard to user-visible data
corruption — see the promotion note above.

Motivating incidents:

1. **Test data leaks dev → real dashboard.** Day 2 smoke-test imports
   (Tester A's CSV scenarios 1-3+6, Tester B's CSVs scenarios 4-5) ran via
   `npm run dev` against the shared userData path, polluting the real
   journal dashboard. Required a manual `.bak` restore from `backups/` to
   recover.

2. **Packaged shadows dev (binary-vs-DB-version mismatch).** After the
   restore, the next launch was the installed packaged build
   (`%LOCALAPPDATA%\Programs\FugaEdge\FugaEdge.exe`, built 2026-05-14 with
   `SCHEMA_VERSION='17'`) instead of the dev branch (`SCHEMA_VERSION='18'`).
   Old binary opened the restored v17 DB, saw "current = target", and
   silently bypassed the v17→v18 migration. No warning that two binaries
   with different schema targets were writing to the same file. ~30 min
   misdiagnosed as a schema-code bug before the file timestamp on the .exe
   gave it away.

3. **Env-var bleed across dev tooling.** Running schema diagnostics under
   Electron's ABI required `ELECTRON_RUN_AS_NODE=1` (the better-sqlite3
   binary is compiled against Electron 33's ABI via the `rebuild:sqlite`
   postinstall, not against system Node). The env var leaked from the
   manual one-liner into a follow-up `npm run dev`, causing Electron to
   launch in Node mode and `require('electron')` to return the path string
   instead of the API namespace — opaque TypeError on
   `protocol.registerSchemesAsPrivileged`. Addressed by
   `npm run inspect:schema` (commit fef5b20, cross-env-scoped), but the
   env-var-bleed-via-shared-shell class is the same dev/test isolation
   theme.

4. **Stale packaged binary corrupts real trade data (2026-05-22).** The
   May-14 v0.1.6 packaged `FugaEdge.exe` was launched and used to import
   LFS trades + journal metadata into the shared
   `%APPDATA%\fugaedge\fugaedge.db`. That binary predates Day 8.5's
   parser-UTC flip, so it stored bare-local-Eastern timestamps; current
   Commit B code reads the column as true UTC and renders trades 12 & 13
   ~5 hours off, permanently. This is the failure mode that promotes the
   ticket — it is no longer just dev friction.

Concept: In dev mode (`!app.isPackaged`), use
`%APPDATA%\fugaedge-dev\fugaedge.db` instead of
`%APPDATA%\fugaedge\fugaedge.db`. Zero-friction separation between dev/test
work and real journal data — `npm run dev` automatically writes to a
separate DB without anyone remembering anything. Directly addresses #1, #2,
and #4 above.

Belt-and-suspenders: add a `FUGAEDGE_DB_PATH` env var override so multiple
fixture sets can each have their own DB when needed (e.g.,
`FUGAEDGE_DB_PATH=...\tester-b-fixtures.db npm run dev`). Also closes future
"I need to test against Tester A's data without touching dev state" scenarios.

Related: any dev script that needs the Electron ABI should be wrapped in a
`cross-env`-scoped npm script (precedent: `inspect:schema` at fef5b20) so
`ELECTRON_RUN_AS_NODE` never leaks into the user's shell session.

Trade-off to think through before implementing: auto-isolation immediately
splits the current real-data dashboard from new dev runs. Anyone who
currently uses `npm run dev` against real data (e.g., journaling between
dev sessions) loses that view unless they launch the packaged installer.
If that's acceptable, ship the auto-isolation. If not, ship the
env-var-override alone first and add auto-isolation later.

Estimated effort: small for the dev-DB isolation itself. A schema-version
mismatch guard on DB open — refuse-or-warn when the binary's
`SCHEMA_VERSION` is older than the DB's `_meta.schema_version`, the silent
gap behind both failure mode #2 and #4 — is a natural companion to scope
during v0.2.1 planning, but it is not part of the original parking-lot
entry and should be decided then.
