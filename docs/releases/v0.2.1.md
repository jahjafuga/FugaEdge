# v0.2.1 — Cross-format dedup fix

Status: draft (no ship date yet)

## What ships in v0.2.1

The v0.2.0 universal-import pipeline let some trades slip past dedup when the same logical fill arrived in two different export formats. Examples:

- A DAS Trades.csv (which has real `TradeID`/`OrderID` columns) and a DAS trades-window CSV (which doesn't) covering the same fills — synthetic `tw-` IDs differ from the real ones, so the existing per-fill-ID hash sees them as different rows.
- A Webull Mobile CSV and a Webull Desktop XLSX covering the same fills — different synthetic-ID prefixes (`wbm-` vs `wbd-`).
- One file emitting `account_name` and one not — the v0.1.6 hash conditionally includes `account_name`, so otherwise-identical fills hash differently.

v0.2.1 adds a second dedup hash (`content_hash`) computed from the intrinsic fill content — symbol, UTC timestamp, side, quantity, price — rather than the per-fill broker IDs. A trade is flagged as duplicate if EITHER the existing ID-based hash OR the new content hash matches an existing row. This catches all three cross-format scenarios above without changing how dedup works for re-imports of the same file (the existing hash still handles those).

## Migration

On first launch after upgrade, FugaEdge:

1. Takes a one-shot backup of your database to `%APPDATA%\fugaedge\backups\fugaedge.db.pre-v0.2.1-content-hash-<timestamp>.bak`.
2. Adds the `content_hash` column to your `trades` table.
3. Computes the content hash for every existing row from the stored fill data and writes it back.

The migration is non-destructive — your existing trade rows are not deleted or modified beyond the new column being populated. Your `exec_hash` column and all other data is untouched. If anything goes wrong during the migration, the backup file is your recovery path (stop the app, copy it over `fugaedge.db`, delete the WAL/SHM sidecars, restart).

## ⚠ If you have historical duplicates from previous imports

If you imported the same trades multiple times before v0.2.1, your existing data may contain duplicates that the new dedup logic can't auto-clean (it doesn't touch existing rows). The migration logs affected trade IDs — see the in-app banner on first launch and remove duplicates manually via the trade detail view.

The banner appears on the Trades page on every launch after the migration until you dismiss it. It tells you the count of detected duplicates. Trade IDs of the affected rows are logged to the FugaEdge log file (Help → Open log folder).

Why we don't auto-clean: a historical duplicate could be intentional (e.g. you deliberately re-recorded a trade with different fills). The safe default is to surface them and let you decide.

## Fixed

- Trade detail chart now labels entry and exit lines as "Avg Entry" and "Avg Exit" to reflect that they're weighted averages from the source data, not single prices.

## What does NOT ship in v0.2.1

- **Real Float data** is still deferred to v0.3.0. The "Shares Out" labeling from v0.2.0 stays. See `docs/plans/v0.3.0-or-later-ideas.md` → "Point-in-Time Float" for the wider-validation findings (2026-05-26) and paid-tier spike (2026-05-27) that drove this decision.
- **Per-trade collision actions** in the data-health banner — clicking a trade ID, batch-deleting selected duplicates, etc. — are deferred to a future "Settings → Data Health" page.

## Known follow-ups for v0.2.1.x hotfix if needed

- A `settings.content_hash_dedup_enabled` toggle is **not** shipped (per design — kill switches advertise risk). If false-positive dedup reports come in, this stub is ready to add in a hotfix.
- Per-trade actions on the collision banner are deferred to v0.3.0.
